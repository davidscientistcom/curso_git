Librerías de mocking

Qué usar, cuándo, y ejemplos mínimos que no fallan.

Nivel 0 — Tu toolkit base (lo que sí o sí debes dominar)

Herramienta Para qué sirve Cuándo usar
unittest.mock Mocks, patching, spies, AsyncMock Siempre (stdlib)
pytest Fixtures, parametrización, plugin ecosystem Si tu proyecto usa pytest (lo habitual)
pytest-mock Fixture mocker (patch limpio y reversible) Cuando quieres patch sin with
monkeypatch (pytest) Parchear env vars, attrs, módulos Config/entorno y patches pequeños
respx / responses Mock de HTTP (httpx/requests) Servicios externos
freezegun Congelar el tiempo Caducidades, tokens, ventanas de tiempo
Idea: el mocking no es una librería; es una técnica. Las librerías sólo te dan herramientas para aplicarla de forma segura.

Nivel 1 — `unittest.mock`: lo esencial en 12 líneas

python — Mock, return_value, side_effect y asserts
from unittest.mock import Mock

def test_core_uses_dependency():
    dep = Mock()

    dep.fetch.return_value = {"value": 10}  # 1) controlas salida

    # 2) tu código usa dep.fetch()
    result = dep.fetch("id-1")["value"] * 2

    assert result == 20
    dep.fetch.assert_called_once_with("id-1")

def test_side_effect_exception():
    dep = Mock()
    dep.fetch.side_effect = TimeoutError("boom")

    try:
        dep.fetch("id")
        assert False
    except TimeoutError:
        assert True

Nivel 1 — Patching: decorator vs context manager

Ambos son correctos. El context manager es más explícito y ayuda a ver el alcance.

python — `patch` como context manager
from unittest.mock import patch

def test_patch_context():
    with patch("app.payments.charge_card") as charge:
        charge.return_value = {"ok": True}

        # aquí llamas a tu función que internamente usa charge_card
        ...

        charge.assert_called_once()
python — `patch` como decorator (más compacto)
from unittest.mock import patch

@patch("app.payments.charge_card")
def test_patch_decorator(charge):
    charge.return_value = {"ok": True}
    ...
    charge.assert_called_once()
Evita: encadenar muchos decorators. Es fácil perder el hilo. Para varios patches, pytest-mock suele quedar más limpio.

Nivel 1 — `autospec`: el antídoto contra mocks “mentirosos”

Sin autospec puedes escribir tests que llaman métodos que no existen. Con autospec el mock respeta la firma real.

python — patch con autospec
from unittest.mock import patch

def test_autospec():
    with patch("app.email.send_email", autospec=True) as send:
        # si send_email(to, subject) requiere 2 args y tú pasas 1, el mock fallará
        ...

Nivel 1 — Async: `AsyncMock` para `await`

python — mockear una coroutine
from unittest.mock import AsyncMock

async def test_async_mock():
    client = AsyncMock()
    client.get.return_value = {"ok": True}

    res = await client.get("/ping")

    assert res["ok"] is True
    client.get.assert_awaited_once()

Nivel 2 — pytest-mock: `mocker.patch` (el patch “limpio”)

La ventaja didáctica de mocker es que el patch se revierte automáticamente al finalizar el test.

python + pytest-mock — patch sin `with`
def test_with_pytest_mock(mocker):
    send = mocker.patch("app.service.send_email", autospec=True)

    from app.service import register_user
    register_user("a@b.com")

    send.assert_called_once()

Nivel 2 — Mock de HTTP: `requests` vs `httpx`

Esto aparece mucho en FastAPI porque es habitual usar httpx para llamar a otros servicios.

`requests`
  • responses es el clásico
  • muy usado en proyectos legacy
`httpx`
  • respx o httpx.MockTransport
  • ideal para FastAPI + async
python — `httpx.MockTransport` (sin dependencias extra)
import httpx

def test_httpx_mocktransport():
    def handler(request: httpx.Request) -> httpx.Response:
        if request.url.path == "/users/1":
            return httpx.Response(200, json={"id": 1, "name": "Ada"})
        return httpx.Response(404, json={"detail": "not found"})

    transport = httpx.MockTransport(handler)
    client = httpx.Client(base_url="https://example", transport=transport)

    res = client.get("/users/1")
    assert res.status_code == 200
    assert res.json()["name"] == "Ada"
Idea: MockTransport es “mocking por diseño”: inyectas el transporte. No parcheas globales.

Nivel 2 — `respx`: mock de rutas HTTP con asserts

Cuando quieres describir “qué endpoints externos se llaman” con claridad, respx queda muy didáctico.

python + respx — route mockeada + assert
import httpx
import respx

def fetch_user(client: httpx.Client, user_id: str) -> dict:
    return client.get(f"https://api.example.com/users/{user_id}").json()

@respx.mock
def test_respx():
    route = respx.get("https://api.example.com/users/1").mock(
        return_value=httpx.Response(200, json={"id": 1, "name": "Ada"})
    )

    client = httpx.Client()
    data = fetch_user(client, "1")

    assert data["name"] == "Ada"
    assert route.called  # se llamó al endpoint esperado

Nivel 2 — `freezegun`: congelar tiempo para tests deterministas

Si generas tokens, calculas expiraciones o comparas fechas, congelar el tiempo es la solución limpia.

python + freezegun — expiry sin flaky tests
from datetime import datetime, timedelta
from freezegun import freeze_time

def expires_at(minutes: int) -> datetime:
    return datetime.utcnow() + timedelta(minutes=minutes)

@freeze_time("2026-03-26 10:00:00")
def test_expiry():
    dt = expires_at(15)
    assert dt.isoformat().startswith("2026-03-26T10:15:00")

Nivel 2 — Mocks específicos de tecnología (opcional pero muy útil)

  • AWS: moto (S3, DynamoDB…)
  • Redis: fakeredis (API compatible en memoria)
  • MongoDB: mongomock (colecciones en memoria)
  • Time: freezegun
  • Replays de HTTP: vcrpy (graba/responde)
Importante: estas librerías ayudan, pero no sustituyen integration tests reales. Úsalas para unit/contract tests, no como “la verdad” del sistema.
