Arquitectura mockeable

La mejor técnica de mocking es: diseñar para que mockear sea trivial.

Nivel 0 — La idea clave: separa lógica de IO

Si tu función mezcla lógica con llamadas externas, te obliga a mockear “media vida”. Si separas bien, la lógica se testea sin mocks y el IO se testea con mocks controlados.

Core (puro)
  • Transformaciones
  • Validaciones
  • Reglas de negocio
  • Sin red, sin DB
Adapters (IO)
  • HTTP clients
  • Repositorios/DB
  • Colas
  • Filesystem
Analogía: el core es la receta. Los adapters son la cocina (hornos, ingredientes, gas). En el examen quieres evaluar si la receta es buena, no si el horno funcionaba.

Nivel 1 — Inyección de dependencias (DI) en Python

DI significa: no crees la dependencia dentro de la función. Recíbela como parámetro. Así puedes pasar un fake/mock en tests.

python — Malo: crear el cliente dentro
import httpx

def get_user_profile(user_id: str) -> dict:
    client = httpx.Client(base_url="https://api.example.com")
    return client.get(f"/users/{user_id}").json()
python — Bien: recibir la dependencia
def get_user_profile(client, user_id: str) -> dict:
    return client.get(f"/users/{user_id}").json()
Por qué esto te salva: en tests puedes pasar un client fake con un método get que devuelve un objeto con .json() sin tocar red.

Nivel 1 — Patrón Repositorio: mockear la DB sin pelearte con SQLAlchemy

En vez de mockear Session.execute() (frágil), define un contrato simple (repositorio) y mockea eso.

python — Contrato y servicio de negocio
from dataclasses import dataclass
from typing import Protocol

class UserRepo(Protocol):
    def get_email(self, user_id: str) -> str | None: ...
    def mark_verified(self, user_id: str) -> None: ...

class EmailSender(Protocol):
    def send(self, to: str, subject: str, body: str) -> None: ...

@dataclass
class VerifyUser:
    repo: UserRepo
    mailer: EmailSender

    def run(self, user_id: str) -> bool:
        email = self.repo.get_email(user_id)
        if email is None:
            return False

        self.repo.mark_verified(user_id)
        self.mailer.send(email, "Cuenta verificada", "OK")
        return True
python + pytest — test sin DB real
from unittest.mock import Mock
from app.usecases import VerifyUser

def test_verify_user_happy_path():
    repo = Mock()
    repo.get_email.return_value = "a@b.com"

    mailer = Mock()

    uc = VerifyUser(repo=repo, mailer=mailer)
    ok = uc.run("usr-1")

    assert ok is True
    repo.mark_verified.assert_called_once_with("usr-1")
    mailer.send.assert_called_once()

Nivel 2 — Pirámide de tests aplicada a mocking

Regla práctica:

  • Unit tests: core puro, casi sin mocks (o mocks de 1 dependencia).
  • Integration tests: DB real local (o SQLite), sin red externa.
  • Contract / API tests: FastAPI con TestClient, mockeando servicios externos.
Si todo es unit con mocks: te falta integración. Los bugs reales suelen estar en el pegamento: serialización, queries, auth, headers, timeouts.

Nivel 2 — Checklist para que un módulo sea “mockeable”

  1. Dependencias externas pasan por parámetros o factories.
  2. No hay singletons globales que se inicialicen al importar.
  3. Los límites de IO están en módulos claramente identificables.
  4. El core no importa librerías de IO (httpx, sqlalchemy, boto3) directamente.
  5. Puedes ejecutar core sin entorno (sin env vars, sin red, sin DB).
