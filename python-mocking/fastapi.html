Mocking en FastAPI

La técnica clave aquí es dependency overrides: mocks sin parchear globales.

Nivel 0 — Qué queremos lograr al testear una API

En tests de FastAPI normalmente quieres:

  • validar status codes y payload
  • validar validación (Pydantic)
  • validar auth y permisos
  • evitar IO real: sin red, sin DB, sin servicios externos
Analogía: FastAPI es la “capa de entrada” de tu app. En tests, la puerta se prueba con cerradura real (validación), pero con actores falsos detrás (dependencias).

Nivel 1 — Patrón estándar: dependencias + override

Patrón base de mocking en FastAPI. No hay magia: sólo DI.

python — App con dependencia (repo)
# app/main.py
from fastapi import FastAPI, Depends

app = FastAPI()

class UserRepo:
    def get_user(self, user_id: str) -> dict | None:
        # DB real aquí (no entraremos en tests)
        ...

def get_repo() -> UserRepo:
    return UserRepo()

@app.get("/users/{user_id}")
def read_user(user_id: str, repo: UserRepo = Depends(get_repo)):
    user = repo.get_user(user_id)
    if user is None:
        return {"detail": "not found"}
    return user
python + pytest — override de dependencia
# tests/test_api.py
from fastapi.testclient import TestClient
from unittest.mock import Mock

from app.main import app, get_repo

def test_read_user_ok():
    repo = Mock()
    repo.get_user.return_value = {"id": "1", "name": "Ada"}

    def override_repo():
        return repo

    app.dependency_overrides[get_repo] = override_repo

    client = TestClient(app)
    res = client.get("/users/1")

    assert res.status_code == 200
    assert res.json()["name"] == "Ada"
    repo.get_user.assert_called_once_with("1")

    # limpieza (importante si hay más tests)
    app.dependency_overrides.clear()
Muy importante: siempre limpia dependency_overrides. Si no, un test contamina a los siguientes.

Nivel 1 — Mock de auth: override del “current user”

En FastAPI, auth suele estar en una dependencia tipo get_current_user. Mockear eso en tests es exactamente lo mismo: override.

python — dependencia de auth
from fastapi import Depends, HTTPException, status

def get_current_user():
    # aquí validarías JWT / sesión
    raise HTTPException(status_code=status.HTTP_401_UNAUTHORIZED)

@app.get("/me")
def me(user = Depends(get_current_user)):
    return user
python + pytest — override para simular usuario
from fastapi.testclient import TestClient
from app.main import app, get_current_user

def test_me_authenticated():
    app.dependency_overrides[get_current_user] = lambda: {"id": "u1", "role": "admin"}

    client = TestClient(app)
    res = client.get("/me")

    assert res.status_code == 200
    assert res.json()["role"] == "admin"

    app.dependency_overrides.clear()

Nivel 2 — FastAPI + async: usar `httpx.AsyncClient` con ASGITransport

Si tu app tiene endpoints async y quieres tests async, este patrón es muy estable.

python — test async de FastAPI sin servidor real
import pytest
import httpx

from app.main import app

@pytest.mark.asyncio
async def test_async_endpoint():
    transport = httpx.ASGITransport(app=app)
    async with httpx.AsyncClient(transport=transport, base_url="http://test") as client:
        res = await client.get("/health")
        assert res.status_code == 200
Clave: no levantas un servidor. El transporte ASGI ejecuta tu app en memoria.

Nivel 2 — Mockear un servicio externo llamado desde el endpoint

Hay dos enfoques (ambos válidos). Empieza por el que requiere menos parches: inyectar cliente o transporte.

Enfoque A (preferido)
  • cliente HTTP como dependencia
  • override en tests
  • sin patch global
Enfoque B (también vale)
  • patch del módulo (unittest.mock)
  • más frágil con imports
python — servicio externo como dependencia
import httpx
from fastapi import Depends

def get_http_client() -> httpx.Client:
    return httpx.Client(base_url="https://billing")

@app.post("/charge")
def charge(payload: dict, client: httpx.Client = Depends(get_http_client)):
    res = client.post("/charge", json=payload)
    return {"ok": res.status_code == 200}
python + pytest — override con MockTransport
from fastapi.testclient import TestClient
import httpx

from app.main import app, get_http_client

def test_charge_ok():
    def handler(request: httpx.Request) -> httpx.Response:
        return httpx.Response(200, json={"id": "ch_1"})

    transport = httpx.MockTransport(handler)

    def override_client():
        return httpx.Client(base_url="https://billing", transport=transport)

    app.dependency_overrides[get_http_client] = override_client

    client = TestClient(app)
    res = client.post("/charge", json={"amount": 10})

    assert res.status_code == 200
    assert res.json()["ok"] is True

    app.dependency_overrides.clear()
Por qué este ejemplo es bueno: se ve DI (override) + un mock “por diseño” (transport). No hay patching de imports.

Nivel 2 — Alternativa: `respx` para interceptar `httpx`

Si prefieres no tocar dependency_overrides, puedes interceptar las llamadas HTTP con respx. Es útil cuando el cliente HTTP se crea dentro de la dependencia y quieres un mock declarativo por rutas.

python + respx — mock por URL (sin override)
import httpx
import respx
from fastapi.testclient import TestClient

from app.main import app

@respx.mock
def test_charge_ok_with_respx():
    # Intercepta la llamada externa a billing
    route = respx.post("https://billing/charge").mock(
        return_value=httpx.Response(200, json={"id": "ch_1"})
    )

    client = TestClient(app)
    res = client.post("/charge", json={"amount": 10})

    assert res.status_code == 200
    assert route.called

Nivel 2 — BackgroundTasks: testear sin ejecutar tareas reales

Si tu endpoint encola tareas (emails, webs hooks), puedes inyectar el “sender” y mockearlo igual. Si es un BackgroundTasks.add_task, normalmente validas que se llamó (spy) en unit tests del handler.

Regla práctica: si la tarea tiene lógica propia, la testéas por separado. El endpoint sólo debería comprobar que la agenda.
