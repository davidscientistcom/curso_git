Mocking de bases de datos

Dos caminos seguros: fakes (in-memory) o integration tests con DB real local.

Nivel 0 — La verdad incómoda: mockear la DB suele ser frágil

Si mockeas cursor.execute o Session.execute, tu test se acopla a detalles internos. El resultado típico: refactors sanos rompen tests y el suite se vuelve frágil.

Regla práctica: mockea el contrato (repositorio) o usa una DB ligera para integración.

Nivel 1 — Opción A: repositorio + mock (unit tests rápidos)

Este patrón se explica muy bien: la DB queda detrás de una interfaz.

python — caso de uso + repositorio
from dataclasses import dataclass
from typing import Protocol

class OrderRepo(Protocol):
    def total_orders_for_user(self, user_id: str) -> int: ...

@dataclass
class UserStats:
    repo: OrderRepo

    def vip_badge(self, user_id: str) -> str:
        n = self.repo.total_orders_for_user(user_id)
        return "VIP" if n >= 50 else "NORMAL"
python + pytest — mock del repo
from unittest.mock import Mock
from app.stats import UserStats

def test_vip_badge():
    repo = Mock()
    repo.total_orders_for_user.return_value = 51

    stats = UserStats(repo=repo)
    assert stats.vip_badge("u1") == "VIP"
    repo.total_orders_for_user.assert_called_once_with("u1")

Nivel 1 — Opción B: fake en memoria (cuando la DB es simple)

Un fake es una implementación real pero “mini”. Es útil porque es tangible y estable.

python — fake repo con dict
class FakeOrderRepo:
    def __init__(self):
        self.orders = {}  # user_id -> count

    def total_orders_for_user(self, user_id: str) -> int:
        return self.orders.get(user_id, 0)

    def add_order(self, user_id: str) -> None:
        self.orders[user_id] = self.orders.get(user_id, 0) + 1

Nivel 2 — Integration tests con SQLAlchemy + SQLite in-memory

Esto elimina “mocks mentirosos”: ejecutas SQL real, pero rápido y local.

Truco importante: SQLite in-memory vive por conexión. Si creas múltiples conexiones, “pierdes” tablas. Por eso se suele reutilizar engine y usar transacciones.
python + pytest — fixture de sesión transaccional
# tests/conftest.py (esqueleto didáctico)
import pytest
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

@pytest.fixture(scope="session")
def engine():
    return create_engine("sqlite+pysqlite:///:memory:", future=True)

@pytest.fixture()
def db_session(engine):
    # 1) conexión + transacción por test
    connection = engine.connect()
    trans = connection.begin()

    Session = sessionmaker(bind=connection, autoflush=False, autocommit=False, future=True)
    session = Session()

    try:
        yield session
    finally:
        session.close()
        trans.rollback()  # 2) deja la DB limpia
        connection.close()

Nivel 2 — FastAPI + DB: override de `get_db()`

El patrón estándar es que FastAPI tenga una dependencia get_db que yield-ea una sesión. En tests, la reemplazas por una sesión de test.

python — dependencia get_db (patrón típico)
# app/db.py
from sqlalchemy.orm import Session

def get_db() -> Session:
    db = ...  # Session real (engine real)
    try:
        yield db
    finally:
        db.close()
python + pytest — override get_db en tests
from fastapi.testclient import TestClient
from app.main import app
from app.db import get_db

def test_api_with_test_db(db_session):
    def override_db():
        yield db_session

    app.dependency_overrides[get_db] = override_db

    client = TestClient(app)
    res = client.get("/users/1")

    assert res.status_code == 200

    app.dependency_overrides.clear()

Nivel 2 — Redis y Mongo: preferir fakes compatibles

Si tu app usa Redis o Mongo, es habitual usar:

  • fakeredis para Redis (API compatible en memoria)
  • mongomock para MongoDB (colecciones en memoria)
Nota: aquí no haces “mocking” a mano; eliges una implementación fake que mantiene la API. Esto reduce la fragilidad.

Nivel 2 — ¿Cuándo usar DB real (Docker/Testcontainers)?

Si dependes de features específicas (Postgres JSONB, locks, índices), SQLite no representa la realidad. En ese caso, usa una DB real local (Docker) y marca esos tests como integración.

Nota: “Mockear Postgres no te da garantías de Postgres”. Si el bug está en el SQL o en el dialecto, necesitas integración.
