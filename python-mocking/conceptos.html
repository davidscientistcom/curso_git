Conceptos fundamentales

Lo mínimo (pero completo) que debes dominar antes de escribir un solo mock.

Nivel 0 — ¿Qué problema resuelve el mocking?

Cuando escribes un test, quieres validar tu lógica, no la red, no la base de datos, y no el proveedor externo que hoy está caído. Mocking es la técnica para controlar dependencias (IO, tiempo, random, servicios externos) y convertir un test en algo:

  • rápido (milisegundos)
  • determinista (mismo input → mismo output)
  • aislado (falla por tu código, no por el entorno)
Analogía: un test es un laboratorio. El mocking es el conjunto de controles para que la temperatura, la luz y los reactivos sean exactamente los que tú decides.

Nivel 0 — Vocabulario: dummy, stub, fake, spy, mock

Se suele mezclar todo. Aquí tienes un mapa mental que evita confusiones:

Stubs / Fakes (estado útil)
  • Stub: devuelve valores predecibles.
  • Fake: implementación simplificada (pero real) en memoria.
  • Ej: una “DB” en memoria con un dict.
Spies / Mocks (verificación)
  • Spy: observa llamadas (argumentos, veces).
  • Mock: reemplaza + verifica contrato.
  • Ej: comprobar que se llamó a send_email(to) 1 vez.
Error típico: usar mocks para todo. Si tu test sólo verifica “se llamó a X”, sin validar resultados, puedes terminar con tests verdes que no detectan bugs reales.

Nivel 1 — Patching: “parchea donde se usa, no donde se define”

Esta es la regla que más dudas genera. El mock no se aplica a “la función original”, se aplica al nombre (símbolo) que tu código usa en ese módulo.

python — Ejemplo mínimo del problema (import) 
# archivo: app/service.py
from app.email import send_email

def register_user(email: str) -> None:
    # ... lógica ...
    send_email(email)  # <-- aquí se usa el símbolo local send_email

# archivo: app/email.py
def send_email(to: str) -> None:
    # llamada real a SMTP / proveedor externo
    ...

Si quieres mockear send_email cuando se ejecuta register_user, debes parchear app.service.send_email (el símbolo dentro del módulo que lo usa).

python + pytest — patch correcto (donde se usa) 
# tests/test_service.py
from unittest.mock import patch

def test_register_user_sends_email():
    with patch("app.service.send_email") as send_email_mock:
        from app.service import register_user
        register_user("a@b.com")

        # assert de comportamiento: se llamó 1 vez con el argumento esperado
        send_email_mock.assert_called_once_with("a@b.com")
Frase útil: “No parcheas la función. Parcheas el nombre que tu módulo tiene en su espacio de nombres.”

Nivel 1 — ¿Qué se debe mockear? (heurística simple)

Mockea lo que sea IO o no determinista:

  • HTTP (requests/httpx) y SDKs externos
  • Base de datos (con repositorios o sesiones aisladas)
  • Tiempo (datetime.now())
  • Aleatoriedad
  • Sistema de ficheros
Regla: tu “core” (lógica pura) debería poder testearse sin mocks. Los mocks viven en los bordes (adapters) del sistema.

Nivel 2 — Assertions que realmente detectan bugs

Un patrón sano es combinar:

  • assert de resultado (lo que devuelve tu función)
  • assert de interacción (qué llamadas hizo a dependencias)
python — Resultado + interacción (ejemplo realista) 
# app/pricing.py
def final_price(base: float, discount_pct: float) -> float:
    return round(base * (1 - discount_pct/100), 2)

# app/checkout.py
from app.pricing import final_price

def checkout_total(items, apply_coupon):
    subtotal = sum(i[1] for i in items)
    pct = apply_coupon(subtotal)  # dependencia (podemos mockearla)
    return final_price(subtotal, pct)
python + pytest — mockear la dependencia y validar total 
from unittest.mock import Mock
from app.checkout import checkout_total

def test_checkout_total_applies_coupon():
    apply_coupon = Mock(return_value=10)

    total = checkout_total(items=[("p1", 100.0)], apply_coupon=apply_coupon)

    # assert de resultado (lo importante)
    assert total == 90.0

    # assert de interacción (evita bugs de integración dentro de la función)
    apply_coupon.assert_called_once_with(100.0)

Nivel 2 — Los 5 anti-patrones a evitar

  1. Mockear lo que no es IO: lógica pura no debería necesitar mocks.
  2. Parchear el sitio incorrecto: test que no controla la dependencia real.
  3. Over-spec: asserts de llamadas demasiado exactos (frágiles).
  4. Testear implementación en vez de comportamiento (se rompe con refactors sanos).
  5. Mocks globales compartidos entre tests (contaminación).
Señal roja: si al cambiar nombres internos (refactor) se rompen 30 tests, tu suite está acoplada a implementación. El mocking está “demasiado cerca”.
