🚀 Nivel 1: Fundamentos Absolutos del Testing (Pytest)

Bienvenidos al laboratorio progresivo de testing. Soy tu Lead QA y hoy vamos a desgranar cómo asegurar la calidad de nuestro código Python. Olvida unittest y su verbosidad heredada de Java; aquí dominaremos pytest, el estándar unánime de la industria.

🔍

Test Automático

Código que ejecuta otro código y verifica que el resultado es el esperado de forma automatizada.

💎

Pytest

Framework moderno de testing en Python. Usa assert nativo en un enfoque declarativo y puro.

⚖️

Assertion

Una afirmación matemática de que una condición debe ser True. Si es False, el test falla.

Ejercicio 01: El Primer Assert y la disección del error

La unidad más fundamental de un test es una función cuyo nombre empieza por test_. Si en su interior un assert falla, Pytest capturará las variables involucradas e imprimirá un reporte (traceback) riquísimo en contexto.

💡
Analogía: Piensa en un assert como el inspector de aduanas. Tú le dices: "El paquete debe pesar 5kg". Él pesa el paquete, y si pesa 6kg, hace saltar una sirena detallando exactamente qué paquete falló y por qué diferencia. Si todo cuadra, te deja pasar en silencio.
python — test_01_assert.py 
def calcular_descuento(precio_base: float, porcentaje: float) -> float:
    """Aplica un descuento. Si el porcentaje es 10 sobre 100, devuelve 90."""
    descuento = precio_base * (porcentaje / 100)
    return precio_base - descuento

# Nuestro primer test
def test_calcular_descuento_basico():
    # 1. Arrange (Preparar)
    precio = 100.0
    descuento = 10.0
    
    # 2. Act (Actuar)
    resultado = calcular_descuento(precio, descuento)
    
    # 3. Assert (Verificar)
    assert resultado == 90.0, "El descuento del 10% sobre 100 debería resultar en 90"

def test_calculo_con_decimales():
    # Pytest nos dirá exactamente los valores si esto falla
    resultado = calcular_descuento(19.99, 5.0)
    # Cuidado: la aritmética flotante en Python puede ser engañosa.
    # Por eso solemos redondear o usar pytest.approx
    assert round(resultado, 2) == 18.99

Para ejecutar esto en la terminal: pytest test_01_assert.py -v

Ejercicio 02: Testeando Excepciones (Context Managers)

No solo testeamos el camino feliz. Un código robusto es aquel que falla de forma controlada y esperada cuando recibe basura. ¿Cómo comprobamos que nuestra función levanta un error sin que ese error detenga la ejecución de la suite de tests?

python — test_02_exceptions.py 
import pytest

def dividir(a: float, b: float) -> float:
    if b == 0:
        raise ValueError("División por cero detectada. Uso indebido.")
    return a / b

def test_dividir_por_cero_levanta_value_error():
    # Usamos pytest.raises como un 'Context Manager' (un bloque with).
    # Todo lo que ocurre dentro de este bloque DEBE levantar un ValueError.
    # Si no lo levanta, el test fallará por "Falso Positivo".
    with pytest.raises(ValueError) as exception_info:
        dividir(10.0, 0.0)
        
    # Podemos inspeccionar el error exacto que se capturó:
    assert "División por cero" in str(exception_info.value)

Ejercicio 03: Agrupación y Descubrimiento (Clases vs Funciones puras)

En Java o en el antiguo unittest de Python, todo test debía residir forzosamente dentro de una Clase que heredaba de TestCase. En Pytest abrazamos el paradigma funcional: una función es suficiente. No obstante, las clases son útiles exclusivamente para agrupar tests relacionados lógicamente.

✅ Funciones puras (Recomendado)
  • Menos boilerplate y sangrías.
  • Fáciles de leer de arriba a abajo.
  • Ideal para tests unitarios aislados.
🏢 Agrupación en Clases (Test Suites)
  • Agrupa escenarios del mismo componente.
  • OBLIGATORIO: El nombre de la clase debe empezar por Test.
  • No deben tener método __init__.
python — test_03_classes.py 
class UserManager:
    def __init__(self):
        self.users = []
        
    def add_user(self, name):
        self.users.append(name)

# Agrupamos todos los tests relacionados con UserManager bajo una clase.
class TestUserManager:
    
    # Cada método es un test si empieza por test_
    def test_anadir_usuario_incrementa_conteo(self):
        manager = UserManager()
        manager.add_user("Alice")
        assert len(manager.users) == 1

    def test_anadir_multiples_usuarios(self):
        manager = UserManager()
        manager.add_user("Alice")
        manager.add_user("Bob")
        assert "Bob" in manager.users

Ejercicio 04: Selección, Filtros e Ignorados (Marcadores)

Cuando tu suite de tests crece a miles de funciones, no querrás ejecutarlos todos cada vez que cambias una coma. Pytest ofrece "Marcadores" (metadata que le pegamos a una función) y filtros de consola.

⚠️
¡Cuidado! Nunca uses if True: return para saltarte un test roto en tu código. Te olvidarás de él. Usa siempre @pytest.mark.skip con una razón formal. Así aparecerá subrayado en amarillo en los reportes de QA.
python — test_04_markers.py 
import pytest
import sys

def funcion_lenta_y_pesada():
    """Simula un proceso que tarda 1 hora."""
    return "completado"

# 1. SKIPPING EXTERNO: Saltamos este test con una justificación.
@pytest.mark.skip(reason="Este test tarda demasiado. Solo para nightly builds.")
def test_muy_lento():
    resultado = funcion_lenta_y_pesada()
    assert resultado == "completado"

# 2. CONDITIONAL SKIP: Saltar solo si se cumple una regla del sistema.
@pytest.mark.skipif(sys.platform == "win32", reason="No funciona en sistemas Windows")
def test_funcionalidad_solo_linux():
    assert True

# 3. EXPECTED FAILURE (xfail): Sabemos que está roto, pero queremos dejar constancia temporal.
@pytest.mark.xfail(reason="Ticket JIRA-123: El proveedor externo cambió la API")
def test_conexion_proveedor_actualmente_rota():
    1 / 0  # Esto fallaría, pero pytest lo marca como "fallo esperado rojo/amarillo"

# 4. MARCAS CUSTOMIZADAS (Para filtrar desde la consola)
@pytest.mark.auth
def test_login_usuario():
    # Ejecutable con: pytest -m auth
    assert "token" != None
