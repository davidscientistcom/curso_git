🏗️ Nivel 2: Inversión de Control con Fixtures

En el nivel anterior iniciábamos variables dentro de cada test. Eso no es escalable. ¿Qué pasa si necesitamos un usuario preparado con permisos de administrador en 50 tests distintos? Aquí entran los Fixtures. Un Fixture inyecta dependencias automáticamente en la firma del test.

💡
Analogía: Un Fixture es como el "Mise en place" de un chef. En lugar de que el chef limpie, corte y prepare las cebollas cada vez que va a cocinar un plato, un ayudante (Pytest) se las entrega ya picadas exactamente como él las pidió, listas en su mesa de trabajo.

Ejercicio 05: Setup inyectado básico

Un fixture es una función decorada con @pytest.fixture que retorna un recurso. Pytest busca si tú pides ese recurso como argumento en el test y lo llama mágicamente.

python — test_05_fixtures_basicos.py 
import pytest

# Definición del Fixture ("El preparador")
@pytest.fixture
def base_de_datos_lista():
    # Simulamos preparar la conexión
    print("\n[SETUP] Conectando a Base de Datos en memoria...")
    return {"users": ["alice", "bob"], "status": "connected"}

# El Test pide el fixture por su nombre exacto en los parámetros
def test_leer_usuarios(base_de_datos_lista):
    # Ya tenemos el objeto listo para usar.
    assert "alice" in base_de_datos_lista["users"]
    assert base_de_datos_lista["status"] == "connected"

def test_no_hay_admin(base_de_datos_lista):
    # El fixture se ejecuta de nuevo para este test (aislamiento total)
    assert "admin" not in base_de_datos_lista["users"]

Ejercicio 06: Ciclos de Vida (Scopes)

¿Qué pasa si el recurso cuesta mucha memoria o tiempo (ejemplo: leer un modelo de IA de 1GB)? Ejecutar el fixture en cada función es ruinoso. Pytest nos permite usar scope.

⚡ scope="function" (Por defecto)
  • Se recrea en cada método de test.
  • Aislamiento máximo, seguridad máxima.
🐢 scope="session"
  • Se crea una sola vez por cada corrida de pytest.
  • Ideal para conexiones pesadas (Postgres/Redis).
python — test_06_scopes.py 
import pytest
import time

# Al usar scope="session", esto solo tardará 2 segundos la PRIMERA vez.
@pytest.fixture(scope="session")
def modelo_ia_pesado():
    print("\n[SESIÓN] Inicializando Red Neuronal...")
    time.sleep(2)  # Simula carga pesada
    return "Modelo V3.0 Cargado"

def test_ia_inferencia_1(modelo_ia_pesado):
    assert "V3.0" in modelo_ia_pesado

def test_ia_inferencia_2(modelo_ia_pesado):
    # Este test no espera 2 segundos. Reutiliza el objeto de memoria global.
    assert "Cargado" in modelo_ia_pesado

Ejercicio 07: Limpieza Garantizada (Teardown con Yield)

¿Qué ocurre cuando creamos una fila en base de datos para testear y necesitamos eliminarla para no manchar el entorno? El fixture puede pausar devolviendo el valor usando yield, y todo el código escrito debajo será el Teardown.

⚠️
¡Cuidado! Nunca uses una función externa genérica de limpieza al final de tu test. Si un assert falla arriba, la ejecución del test se corta inmediatamente y nunca llegaría a tu limpieza. El fixture con yield garantiza la limpieza *incluso si hay un error brutal* en el test.
python — test_07_teardown_yield.py 
import pytest

# Simulamos crear un archivo y borrarlo temporalmente
@pytest.fixture
def archivo_temporal():
    # --- BLOQUE SETUP ---
    archivo = "temp_config.txt"
    print(f"\n[SETUP] Escribiendo archivo: {archivo}")
    open(archivo, "w").write("config=true")
    
    # Pausamos y devolvemos la respuesta al test.
    yield archivo
    
    # --- BLOQUE TEARDOWN ---
    # Pytest retomará esta línea cuando el test termine (ya sea por success o porque falló)
    import os
    print(f"\n[TEARDOWN] Borrando archivo: {archivo}")
    if os.path.exists(archivo):
        os.remove(archivo)

def test_leer_archivo_config(archivo_temporal):
    contenido = open(archivo_temporal).read()
    assert "config=true" in contenido
    # Imagina que forzamos un fallo:
    # assert False --> Pytest ejecutará el os.remove() de arriba igualmente.

Ejercicio 08: El Archivo Mágico (conftest.py)

Poner los fixtures en el mismo archivo del test está bien al principio. Pero pronto querremos que todos los tests de nuestra app tengan acceso a archivo_temporal. Para ello, creamos un archivo en la raíz de nuestro directorio de tests llamado conftest.py.

Cualquier fixture definido en un conftest.py estará automáticamente disponible globalmente para todos los test que cuelguen de ese directorio, sin necesidad de importarlo.

python — conftest.py 
# Ubicación: /tests/conftest.py
import pytest

@pytest.fixture(scope="session")
def engine_postgres():
    """Conexión global compartida por toda la suite de pruebas."""
    return "sqlite:///:memory:"  # Usamos sqlite en memoria como mockup
