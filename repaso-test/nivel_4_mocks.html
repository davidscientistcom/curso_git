🎭 Nivel 4: Dobles de Prueba (Mocks y Stubs)

Llegamos al terreno más pantanoso del QA. Un Test Doble es un objeto que finge ser otro objeto durante un test. Si abusas de los dobles, tendrás una suite de pruebas que SIEMPRE pasa en verde, incluso cuando tu código de producción está completamente roto.

🪆

Dummy

Objetos vacíos pasados por puro trámite para llenar argumentos, pero que nunca se usan.

☎️

Mock

Un objeto que registra CÓMO ha sido llamado (con qué argumentos, cuántas veces).

🤖

Stub

Un objeto al que le programas respuestas "enlatadas" (devuelve 200 OK pase lo que pase).

🕵️

Spy / Fake

Spies envuelven objetos reales. Fakes tienen lógica real simplificada (ej. DB en memoria).

Ejercicio 12: El peligro del Mock (Cuándo NO Mockear)

La regla de oro es: "No mockees lo que no te pertenece". Si usas la librería requests de terceros, no mockees el objeto requests.get asumiendo cómo se comportará por dentro. Si la librería se actualiza y cambia su comportamiento, tu test seguirá pasando engañosamente.

⚠️
Antipatrón: Mockear el ORM (como SQLAlchemy o Django ORM). Los ORMs son complejos; si los mockeas, estás testeando que sabes escribir mocks, no que tus consultas a BD funcionan. Usa una BD de test en su lugar.

Ejercicio 13: Stubs Básicos (Monkeypatching en Pytest)

Antes de usar librerías complejas, pytest ofrece el fixture integrado monkeypatch. Nos permite sobrescribir variables de entorno o funciones globales de forma segura (pytest lo restaura automáticamente tras el test).

python — test_13_monkeypatch.py 
import os
import pytest

def obtener_ruta_config():
    # En producción esto podría ser /etc/myapp/config.yaml
    return os.environ.get("APP_CONFIG_PATH", "/default/path/config.yaml")

def test_leer_ruta_customizada(monkeypatch):
    # 1. Usamos monkeypatch para forzar (Stub) una variable de entorno
    monkeypatch.setenv("APP_CONFIG_PATH", "/tmp/test/config.yaml")
    
    # 2. Act
    ruta = obtener_ruta_config()
    
    # 3. Assert
    assert ruta == "/tmp/test/config.yaml"
    # No hace falta limpiar, monkeypatch restaurará el os.environ original aquí.

Ejercicio 14: El objeto MagicMock (Rastreo)

De la librería estandar unittest.mock usamos MagicMock cuando queremos verificar CÓMO se interactuó con una dependencia. Se usa en Arquitectura Limpia para verificar que el Caso de Uso llamó al Repositorio correctamente.

python — test_14_magicmock.py 
from unittest.mock import MagicMock

class ServicioEmail:
    def enviar(self, destinatario, asunto):
        print("Enviando un email real. ¡Cuesta dinero!")

def notificar_usuario(email_service, email):
    # Lógica de dominio brillante
    asunto = f"Bienvenido {email}"
    email_service.enviar(email, asunto)

def test_notificar_usuario_llama_al_servicio():
    # 1. Arrange: Creamos un Doble
    mock_email = MagicMock()
    
    # 2. Act: Inyectamos el doble en lugar del servicio real (Dependency Injection)
    notificar_usuario(mock_email, "test@test.com")
    
    # 3. Assert (Verificación de interacciones)
    # Comprobamos que el método 'enviar' del mock fue invocado.
    assert mock_email.enviar.called
    
    # Comprobamos exactamente con qué argumentos
    mock_email.enviar.assert_called_once_with("test@test.com", "Bienvenido test@test.com")

Ejercicio 15: Forzar retornos (return_value)

A un Mock también podemos asgnarle un comportamiento de Stub estático a través de su atributo return_value.

python — test_15_return_value.py 
from unittest.mock import MagicMock

class ServicioPagos:
    def cobrar(self, importe):
        raise NotImplementedError("Conecta con Stripe real")

def procesar_carrito(servicio_pagos, total):
    if servicio_pagos.cobrar(total) == "OK":
        return "Pedido Completado"
    return "Error en pago"

def test_pago_exitoso():
    # 1. Arrange
    mock_pagos = MagicMock()
    # Programamos la respuesta "enlatada" del mock
    mock_pagos.cobrar.return_value = "OK"
    
    # 2. Act
    resultado = procesar_carrito(mock_pagos, 50.0)
    
    # 3. Assert
    assert resultado == "Pedido Completado"

Ejercicio 16: Spies y Side Effects (Dinámicos y Errores)

¿Qué pasa si queremos simular que el servicio de pagos lanza un Timeout? O peor aún... ¿que el servicio lance "Fallo" la primera vez, pero "OK" la segunda vez (simulando un reintento)? Usamos side_effect.

python — test_16_side_effect.py 
from unittest.mock import MagicMock
import pytest

def test_pago_con_timeout():
    mock_pagos = MagicMock()
    # side_effect levanta una excepción real si le pasamos una clase de Error
    mock_pagos.cobrar.side_effect = TimeoutError("Stripe no responde")
    
    with pytest.raises(TimeoutError):
        # El código a testear explotará, tal como queríamos simular
        mock_pagos.cobrar(100.0)

def test_pago_falla_luego_acierta():
    mock_pagos = MagicMock()
    # Al pasar una lista, cada llamada secuencial recibe el siguiente valor
    mock_pagos.cobrar.side_effect = [ConnectionError, "OK"]
    
    # Llamada 1: Explotaría si no la controlamos
    with pytest.raises(ConnectionError):
        mock_pagos.cobrar(100.0)
        
    # Llamada 2: Funciona
    assert mock_pagos.cobrar(100.0) == "OK"

Ejercicio 17: Mock de Red (No uses HTTP real)

Nunca, bajo ningún concepto, tus tests unitarios deben hacer una petición HTTP real hacia internet. Serán lentos, intermitentes (flaky tests) y en un pipeline bloqueado sin internet, fallarán. Usamos un módulo como responses o httpx-mock para interceptar peticiones del sistema.

❌ Mockear la librería (Malo)
  • mock_requests.get.return_value = ...
  • La librería puede repactar por debajo.
✅ Mockear a nivel de socket (Bueno)
  • Librerías como responses.
  • Interceptan peticiones HTTP a bajo nivel y simulan el tráfico real.
python — test_17_responses.py 
import requests
import responses # pip install responses

@responses.activate # Decorador que enjaula la red
def test_peticion_api_mockeada():
    # Programamos el router HTTP virtual para interceptar la llamada
    responses.add(
        responses.GET,
        "https://api.github.com/users/deivit",
        json={"login": "deivit", "company": "NeuralBrain"},
        status=200
    )
    
    # Ejecutamos el código real, convencido de que habla con GitHub
    respuesta = requests.get("https://api.github.com/users/deivit")
    data = respuesta.json()
    
    assert data["company"] == "NeuralBrain"
    assert respuesta.status_code == 200
    # responses se asegura de que la petición no salió a internet
