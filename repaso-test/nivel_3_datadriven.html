🧬 Nivel 3: Acelerando Datos (Data-Driven Testing)

La regla DRY (Don't Repeat Yourself) también aplica a los tests. Si tienes una función que valida emails y quieres probar 15 emails inválidos, escribir 15 funciones test_email_invalido_1(), test_email_invalido_2(), etc. es un antipatrón masivo. Para esto usamos Data-Driven Testing mediante la parametrización.

💡
Analogía: Piense en la parametrización como una fábrica de coches. En lugar de construir una fábrica entera nueva para probar cómo resiste un choque un coche rojo, azul o verde, mantienes la misma máquina de choque (el test) y simplemente le vas pasando coches de distinto color por la cinta transportadora (los parámetros).

Ejercicio 09: Parametrización Lineal (El decorador mágico)

Con @pytest.mark.parametrize definimos una cadena con los nombres de las variables, y luego una lista de tuplas con los valores. Pytest generará un test distinto e independiente por cada tupla.

python — test_09_parametrize.py 
import pytest

def es_palindromo(palabra: str) -> bool:
    s = palabra.lower().replace(" ", "")
    return s == s[::-1]

# Definimos 2 variables: "palabra" y "esperado"
# Pasamos 3 escenarios distintos. ¡Pytest ejecutará 3 tests!
@pytest.mark.parametrize("palabra, esperado", [
    ("Reconocer", True),
    ("Dabale arroz a la zorra el abad", True),
    ("Python", False)
])
def test_es_palindromo_variaciones(palabra, esperado):
    # La lógica del test es idéntica, solo cambian los datos.
    assert es_palindromo(palabra) == esperado

Ejercicio 11: Parametrizando Fixtures

¿Qué ocurre si la variabilidad no está en los datos matemáticos, sino en la infraestructura subyacente? Por ejemplo: queremos pasar todos nuestros tests de base de datos usando SQLite primero, y luego usando PostgreSQL pura para asegurar compatibilidad.

Aquí parametrizamos el Fixture usando params=[], y extraemos el valor usando el objeto mágico request.param.

📊 Parametrizar Funciones
  • Usa @pytest.mark.parametrize.
  • Ideal para datos simples (Strings, ints, tuplas).
⚙️ Parametrizar Fixtures
  • Usa @pytest.fixture(params=[...]).
  • Ideal para motores, conexiones, clientes HTTP.
python — test_11_fixture_param.py 
import pytest

class FakeDB:
    def __init__(self, motor):
        self.motor = motor
    def is_alive(self):
        return True

# El fixture se ejecutará 2 veces: una para postgres, otra para mysql.
# Por tanto, todos los tests que pidan 'db_connection' se ejecutarán 2 veces.
@pytest.fixture(params=["postgres", "mysql"])
def db_connection(request):
    print(f"\n[SETUP DB] Iniciando motor: {request.param}")
    
    db = FakeDB(request.param)
    
    yield db
    
    print(f"\n[TEARDOWN DB] Cerrando motor: {request.param}")

def test_bd_is_alive_en_ambos_motores(db_connection):
    # Este único test resultará en verdad en la ejecución de:
    # test_bd_is_alive_en_ambos_motores[postgres]
    # test_bd_is_alive_en_ambos_motores[mysql]
    assert db_connection.is_alive() is True
    assert db_connection.motor in ["postgres", "mysql"]
