M4 — Testing Async con pytest

Cómo mockear correctamente funciones asíncronas, testear bases de datos simuladas y usar pytest-asyncio.

1. Configuración de pytest-asyncio

Por defecto, pytest no sabe qué hacer si le pasas un test definido como async def (te dará un coroutine warning). Necesitamos la librería pytest-asyncio.

🔧
Configuración Crítica: En el archivo pyproject.toml o pytest.ini, añade asyncio_mode = "auto". Esto ahorra tener que poner @pytest.mark.asyncio encima de cada maldito test.

2. Fixtures Asíncronas (async fixtures)

A menudo necesitamos establecer una conexión a base de datos de tests antes de que empiece un test, y cerrarla al acabar.

python — conftest.py
import pytest
import asyncio

# Usar async def en una fixture requiere pytest-asyncio
# En vez de 'return', usamos 'yield' para poder limpiar recursos al terminar.
@pytest.fixture
async def db_connection():
    # ---- SETUP Phase ----
    print("Conectando a DB...")
    await asyncio.sleep(0.1) 
    fake_db = {"users": [1, 2, 3]}
    
    # Entregamos la DB al test
    yield fake_db 
    
    # ---- TEARDOWN Phase (se ejecuta tras acabar el test) ----
    print("Cerrando conexión DB...")
    await asyncio.sleep(0.1)
    fake_db.clear()

# El test inyecta la fixture de forma normal
async def test_database_has_users(db_connection):
    assert len(db_connection["users"]) == 3

3. El arte de mockear Coroutines: AsyncMock

Este es el punto que más dolores de cabeza da. Si tienes una función síncrona, usas MagicMock. Si tienes una función asíncrona y la mockeas con MagicMock, el test fallará con un error diciendo "objeto Mock no se puede hacer await".

La solución en Python 3.8+ viene de fábrica en la librería estándar: unittest.mock.AsyncMock.

python — test_mocking.py
from unittest.mock import patch, AsyncMock
import pytest

# --- Código de producción que queremos testear ---
async def check_user_exists(api_client, user_id):
    # Imaginemos que esta llamada tarda 5 segundos y cuesta $0.05 a la API..
    # ¡NO QUEREMOS LLAMARLA DE VERDAD EN UN TEST UNITARIO!
    response = await api_client.get_user(user_id)
    return response.get("status") == "active"

# --- El test usando AsyncMock ---
@pytest.mark.asyncio
async def test_user_exists_mock():
    mock_client = AsyncMock()
    
    # Decimos: "Cuando se haga await sobre get_user(123), devuelve este dict"
    mock_client.get_user.return_value = {"status": "active", "name": "Pepe"}
    
    # El código de prod hará "await mock_client.get_user(123)", lo que 
    # retornará instantáneamente lo que dijimos arriba.
    resultado = await check_user_exists(mock_client, 123)
    
    assert resultado is True
    # Verificamos que nuestro código sí llamó al método subyacente
    mock_client.get_user.assert_called_once_with(123)
💡
¿Sabías que...? Si usas @patch('modulo.api_client.get_user'), a partir de Python 3.8 python detecta si la función original era async def y automáticamente la reemplaza con un AsyncMock en lugar de un MagicMock normal. ¡Magia!

4. Race Condition con hilos en Python

⚠️
¡Cuidado! En Python, la mayoría de ejemplos de "race condition" con hilos no fallan porque el GIL (Global Interpreter Lock) impide que dos hilos ejecuten bytecode a la vez.

¿Por qué el ejemplo clásico self.value += 1 NO falla?
La operación self.value += 1 es tan rápida que el GIL no se libera entre el momento de leer y el de escribir el valor. Por eso, aunque lances 10 hilos, el resultado suele ser siempre correcto (no se ve la race condition).

¿Cómo forzar la race condition?
Hay que separar explícitamente el leer del escribir y meter un time.sleep() diminuto entre medias. Así el planificador de hilos puede "interrumpir" y otro hilo pisa el valor.
python — Ejemplo de race condition real
import threading
import time

# 1. LA IMPLEMENTACIÓN INSEGURA
class UnsafeCounter:
    def __init__(self):
        self.value = 0
        
    def increment(self):
        # EXPLICACIÓN DEL BUG:
        # Esta operación parece una sola línea, pero son 3 pasos separados:
        #   Paso 1 → Leer self.value y guardarlo en una variable temporal
        #   Paso 2 → Sumar 1 a esa variable temporal  
        #   Paso 3 → Escribir el resultado de vuelta en self.value
        #
        # Si entre el Paso 1 y el Paso 3 el sistema operativo "pausa" 
        # este hilo y ejecuta otro, ese otro hilo leerá el valor VIEJO
        # y ambos escribirán el mismo resultado → perdemos incrementos.
        #
        # Con time.sleep() forzamos ese "pause" para que sea siempre visible.
        
        current = self.value        # Paso 1: LEER
        time.sleep(0.00001)         # Simulamos trabajo / forzamos context switch
        self.value = current + 1    # Paso 3: ESCRIBIR (con valor ya desactualizado)

# 2. EL TEST (TDD) — Este test DEBE fallar para demostrar la race condition
def test_unsafe_counter_race_condition():
    counter = UnsafeCounter()
    
    # Función que cada hilo ejecutará 50 veces (menos iteraciones por el sleep)
    def worker():
        for _ in range(50):
            counter.increment()
            
    # Creamos 10 hilos
    threads = [threading.Thread(target=worker) for _ in range(10)]
    
    # Arrancamos los hilos y esperamos a que terminen (.join)
    for t in threads: t.start()
    for t in threads: t.join()
    
    total_esperado = 10 * 50  # = 500
    
    print(f"Esperado: {total_esperado}")
    print(f"Resultado real: {counter.value}")
    print(f"Incrementos PERDIDOS: {total_esperado - counter.value}")
    
    # Este assert FALLARÁ porque habrá race conditions.
    # En un contexto de curso, esto es exactamente lo que queremos demostrar.
    assert counter.value < total_esperado, (
        f"RACE CONDITION DEMOSTRADA: esperábamos {total_esperado} "
        f"pero obtuvimos {counter.value}. "
        f"Se perdieron {total_esperado - counter.value} incrementos."
    )
