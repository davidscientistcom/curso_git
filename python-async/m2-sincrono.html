M2 — Python Síncrono con TDD

Entendiendo threads, procesos, race conditions y cómo testear código síncrono concurrente para ver por qué preferimos asyncio.

1. El Riesgo de los Threads (Race Conditions)

El verdadero problema de threading no es la concurrencia en sí, sino el acceso concurrente a estados compartidos mutables. Cuando varios hilos pueden leer y modificar la misma variable al mismo tiempo, aparecen los bugs más difíciles de detectar: race conditions.

⚠️
En Python, una operación aparentemente atómica como contador += 1 son en realidad múltiples instrucciones de bytecode a bajo nivel (LOAD, ADD, STORE). El sistema operativo puede pausar tu hilo en cualquier momento, incluso entre esas instrucciones, permitiendo que otro hilo modifique el mismo estado y generando resultados inesperados.

Pruebas y Código: Demostración TDD del problema

Vamos a escribir un test que demuestre cómo un contador inseguro falla debido a condiciones de carrera, y luego crearemos un contador seguro usando un Lock.

python — test_threads.py
import threading

# 1. LA IMPLEMENTACIÓN INSEGURA
class UnsafeCounter:
    def __init__(self):
        self.value = 0
        
    def increment(self):
        # Separar lectura y escritura para forzar la race condition
        current = self.value
        import time; time.sleep(0.00001)  # Forzamos cambio de contexto aquí
        self.value = current + 1

# 2. EL TEST (TDD)
def test_unsafe_counter_race_condition():
    counter = UnsafeCounter()
    
    def worker():
        for _ in range(50):
            counter.increment()
            
    threads = [threading.Thread(target=worker) for _ in range(10)]
    
    for t in threads: t.start()
    for t in threads: t.join()
        
    total_esperado = 10 * 50  # = 500
    print(f"Esperado: {total_esperado}")
    print(f"Resultado real: {counter.value}")
    print(f"Incrementos PERDIDOS: {total_esperado - counter.value}")
    assert counter.value < total_esperado, (
        f"RACE CONDITION DEMOSTRADA: esperábamos {total_esperado} "
        f"pero obtuvimos {counter.value}. "
        f"Se perdieron {total_esperado - counter.value} incrementos."
    )

# 3. LA IMPLEMENTACIÓN SEGURA
class SafeCounter:
    def __init__(self):
        self.value = 0
        self._lock = threading.Lock() # Creamos un candado
        
    def increment(self):
        # Con 'with', el hilo coge la llave del candado. Al salir, la devuelve.
        with self._lock:
            self.value += 1

# 4. EL TEST VERDE (TDD)
def test_safe_counter():
    counter = SafeCounter()
    
    def worker():
        for _ in range(100_000):
            counter.increment()
            
    threads = [threading.Thread(target=worker) for _ in range(10)]
    for t in threads: t.start()
    for t in threads: t.join()
        
    # AHORA SÍ: SIEMPRE será 1,000,000 exacto.
    assert counter.value == 1_000_000
    print(f"Resultado final SEGURO: {counter.value} ✅")

2. Arreglando el código con Locks

Para arreglar esto, necesitamos que el acceso a self.value += 1 sea exclusivo. Para eso usamos threading.Lock().

# 3. LA IMPLEMENTACIÓN SEGURA
class SafeCounter:
    def __init__(self):
        self.value = 0
        self._lock = threading.Lock() # Creamos un candado
        
    def increment(self):

            self.value += 1

            counter.increment()
            
    threads = [threading.Thread(target=worker) for _ in range(10)]

3. Ejecutando Tareas Masivas: ThreadPoolExecutor

Manejar listas de threads manualmente (crear, `.start()`, `.join()`) es arcaico y muy propenso a errores. En Python moderno (incluso sin asyncio), usamos la API de concurrent.futures.

python — concurrent.futures
from concurrent.futures import ThreadPoolExecutor
import time

def fake_download(url_id: int) -> str:
    time.sleep(0.5) # Simulamos I/O (espera de red)
    return f"Data del id {url_id}"

def test_threadpool_executor():
    start = time.time()
    
    # ThreadPoolExecutor gestiona automáticamente los recursos por ti.
    # Si pones max_workers=5, lanza de 5 en 5.
    with ThreadPoolExecutor(max_workers=5) as executor:
        # El método 'map' ejecuta la función concurrentemente con todos los inputs.
        results = list(executor.map(fake_download, range(10)))
        
    duration = time.time() - start
    
    # 10 tareas de 0.5 seg con 5 workers = 1 segundo en total!
    assert len(results) == 10
    print(f"Terminado en {duration:.2f}s")
⚖️
¿Por qué pasarnos a Asyncio si ThreadPool es tan cómodo?
Porque un thread consume MUCHÍSIMA RAM (aprox 8 MB por hilo). Si necesitas manejar 10,000 conexiones concurrentes (ej: WebSockets chat), ThreadPool moriría por memoria. Asyncio puede manejar 100,000 conexiones en 1 solo hilo y muy poca RAM.
