M6 — Ecosistema Async

No todo en Python está adaptado para asyncio. Conoce las librerías adecuadas y cómo mezclar código síncrono antiguo con moderno.

1. El Gran Peligro de Bloquear el Event Loop

requests.get(), time.sleep(), y las queries normales de psycopg2 o SQLAlchemy sync son enemigos mortales del Event Loop.

Si llamas a una función bloqueante (síncrona) dentro de una coroutine, TODO el framework asyncio se detiene. Las demás peticiones no se procesan hasta que termine. ¡NUNCA uses time.sleep() o requests en llamadas async!

3. FastAPI: `def` vs `async def`

FastAPI es brillante porque soporta ambos mundos, pero equivocarse puede tumbar tu servidor en producción.

Vía `def` normal

FastAPI enviará esta petición a un ThreadPool síncrono externo. Usa esto si tu función hace matemáticas pesadas o utiliza SQLAlchemy síncrono.

@app.get("/")
def sync_route():
  return db.query(...)

Vía `async def`

FastAPI la ejecuta en el Event Loop principal. Usa esto para Asyncio puro. PERO ten cuidado, si bloqueas aquí, bloqueas toda la aplicación web.

@app.get("/")
async def async_route():
  return await db.a_query(...)

4. Mezclando Mundos: `asyncio.to_thread`

¿Qué pasa si necesitas usar una librería antigua (ej. boto3 de AWS o pandas) dentro de tu aplicación asyncio? Usamos asyncio.to_thread() para empujarla fuera del Event Loop principal.

python — Puente Sync-Async
import asyncio
import time

# Una función de terceros que NO PODEMOS cambiar (es síncrona y pesada)
def libreria_legacy_bloqueante():
    print("Iniciando tarea legacy pesada...")
    time.sleep(2) # Va a bloquear
    return "Resultado Legacy"

async def main():
    # ❌ MAL: Bloquearía todo el Event Loop durante 2s
    # res = libreria_legacy_bloqueante() 
    
    # ✅ BIEN: Empujamos la tarea a un ThreadPool gestionado por asyncio.
    # Así devolvemos un 'awaitable' que no congela el loop central.
    res = await asyncio.to_thread(libreria_legacy_bloqueante)
    
    print(res)
