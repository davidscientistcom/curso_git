M1 — GIL & Modelo de Ejecución

Entendiendo las entrañas de CPython, el famoso GIL, y la diferencia entre Concurrencia, Paralelismo y Asincronía.

¿Qué es un Proceso?

Cuando ejecutas un programa Python (por ejemplo, python mi_script.py), el sistema operativo crea un PROCESO. Un proceso es una "caja" aislada que contiene:

  • Su propio espacio de memoria (variables, objetos, datos)
  • Su propio contador de programa (qué línea está ejecutando)
  • Sus propios recursos (ficheros abiertos, conexiones de red)

Los procesos son INDEPENDIENTES entre sí. Si un proceso falla o se corrompe, los demás procesos no se ven afectados. La comunicación entre procesos (IPC - Inter-Process Communication) es costosa porque hay que copiar datos entre "cajas" aisladas.

💡
Ejemplo real: Cuando tienes Chrome abierto con 10 pestañas, cada pestaña es un proceso separado. Si una pestaña falla, las demás siguen funcionando.

¿Qué es un Hilo (Thread)?

Dentro de un proceso puede haber múltiples HILOS (threads). Los hilos son "subunidades de ejecución" que viven DENTRO del mismo proceso. Características clave:

  • COMPARTEN la memoria del proceso (pueden leer y escribir las mismas variables)
  • Son más ligeros que los procesos (creación más rápida, menos recursos)
  • La comunicación entre hilos es directa y barata (memoria compartida)
  • PELIGRO: al compartir memoria, pueden "pisarse" mutuamente → race conditions
🍳
Analogía: El proceso es una cocina. Los hilos son los cocineros dentro de esa cocina. Todos comparten los mismos ingredientes (memoria compartida). Si dos cocineros intentan usar la misma sartén a la vez sin coordinarse → ¡desastre! (race condition).

Proceso vs Hilo — Cuándo usar cada uno

Característica Proceso Hilo (Thread)
Memoria Aislada (propia) Compartida con otros hilos
Velocidad de creación Lento (~50ms) Rápido (~1ms)
Comunicación Costosa (IPC, colas, pipes) Barata (memoria compartida)
Fallo de uno No afecta a otros procesos Puede tumbar todo el proceso
En Python Evita el GIL → CPU-bound Limitado por el GIL → I/O-bound
Librería Python multiprocessing threading
Cuándo usarlo Cálculo intensivo (ML, vídeo) Esperas de I/O (APIs, archivos)

El GIL — Por qué Python es diferente

Python (CPython, la versión estándar) tiene una restricción llamada GIL (Global Interpreter Lock — Bloqueo Global del Intérprete).

El GIL es un mecanismo de seguridad que garantiza que, en cualquier momento, SOLO UN HILO puede ejecutar código Python a la vez, aunque tengas 8 núcleos de CPU disponibles.

¿Por qué existe? Para proteger las estructuras internas del intérprete de Python de la corrupción por acceso concurrente. Es una decisión de diseño histórica (Python 1991) que facilitó mucho la implementación.

¿Qué implica para nosotros?

  • Threading en Python NO paraleliza código que usa mucha CPU
  • Threading en Python SÍ es útil para operaciones de I/O (cuando un hilo espera una respuesta de red, el GIL se libera y otro hilo puede ejecutar)
  • Para paralelismo real de CPU → usar multiprocessing o asyncio
🧩
IMPORTANTE: Python 3.13 introduce el modo "free-threaded" (sin GIL) como experimento, pero aún no es el comportamiento por defecto.

1. El Global Interpreter Lock (GIL)

Antes de hablar de asincronía, necesitamos entender a nuestro "amigo" y cuello de botella favorito: el GIL.

⚠️
¿Qué es el GIL? Es un mecanismo tipo mutex (candado) que protege el acceso a los objetos de CPython, previniendo que múltiples hilos de ejecución ejecuten bytecodes de Python simultáneamente.

Analogía Inmobiliaria 🏠

Imagina que CPython es una obra de construcción y tienes 4 obreros (hilos - threads). Sin embargo, solo hay un martillo (el GIL). Aunque tengas 4 obreros, solo el obrero que tiene el martillo en la mano puede clavar clavos. Los demás tienen que esperar.

¿Es python lento por esto? Depende. Si tu programa hace muchas matemáticas (clavar clavos todo el tiempo), sí, el GIL te frena. Pero si tu programa hace muchas peticiones de red (pedir materiales y esperar a que lleguen), el obrero suelta el martillo mientras espera, permitiendo que otro obrero trabaje. Aquí es donde brilla el enfoque asíncrono.

2. CPU-Bound vs I/O-Bound

💻 CPU-Bound

El cuello de botella es el procesador. Cálculos matemáticos, procesamiento de imágenes, encriptación.

El GIL impacta negativamente. Requiere multiprocessing (contratar a 4 obreros distintos con 4 martillos distintos).

🌐 I/O-Bound

El cuello de botella es esperar. Peticiones de red, lectura de bases de datos, leer disco.

El GIL NO impacta negativamente. Podemos usar threading o asyncio, porque el hilo "suelta el candado" mientras espera.

3. Concurrencia vs Paralelismo vs Asincronía

Estos 3 términos suelen confundirse. Veámoslo de forma sencilla:

  • Concurrencia: Hacer malabares con varias tareas (avanzas un poquito en cada una). Puede ser con 1 o más núcleos.
  • Paralelismo: Hacer realmente varias cosas AL MISMO TIEMPO EXACTO (requiere múltiples núcleos físicos). En Python, esto se hace con multiprocessing.
  • Asincronía: Una forma específica de concurrencia donde un solo hilo decide cuándo saltar de tarea en tarea, usando la instrucción de "espera" (await) para no bloquearse e irse a hacer otra cosa útil mientras tanto.
💡
La Regla de Oro en Python:
1. ¿Tareas matemáticas fuertes? Usa multiprocessing.
2. ¿Peticiones a red, API, BD o disco? Usa asyncio (y en menor medida threading). Daremos prioridad a asyncio por ser más seguro y predecible.
python — Detectando si algo es bloqueante (Ejemplo mental)
# Esto es una operación I/O-Bound. El GIL se libera automáticamente.
# Perfecto para asyncio o threading.
response = requests.get("https://api.github.com")

# Esto es una operación CPU-Bound. El GIL se mantiene bloqueado.
# Congelará todo el programa. Necesita multiprocessing.
total = sum(i * i for i in range(10_000_000))
