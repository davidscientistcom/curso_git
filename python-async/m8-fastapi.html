M8 — FastAPI: Entendiendo el Motor

Cómo FastAPI decide ejecutar tu código. El impacto crítico de def vs async def en aplicaciones de producción.

1. El Secreto de FastAPI (Starlette)

FastAPI (construido sobre Starlette) es fundamentalmente un framework asíncrono que corre dentro de un Event Loop (usualmente vía uvicorn). Sin embargo, fue diseñado para ser accesible y permite definir rutas (endpoints) síncronas tradicionales.

⚖️
Frente a una petición, FastAPI inspecciona tu función. Si pones async def, la ejecuta directamente en el hilo principal (Event Loop). Si pones def, la envía a un ThreadPool (hilos secundarios) gestionado por anyio.

2. Escenarios Clave: Qué usar y Cuándo

Escenario A: I/O Asíncrono

Estás usando httpx, asyncpg, o Motor (MongoDB async).

Usa: async def

@app.get("/")
async def read_data():
  return await db.fetch_all()

Escenario B: I/O Síncrono (típico)

Estás usando SQLAlchemy clásico, psycopg2 o requests.

Usa: def

@app.get("/")
def read_data():
  return db.query(User).all()

3. Los Errores que Tumban Servidores

Veamos los dos antipatrones más comunes que colapsarán la capacidad (throughput) de tu API.

❌ Error 1: Código bloqueante en una función `async def`

Este es el peor error posible. Si usas async def, FastAPI confía en ti y te da el hilo principal. Si haces time.sleep() o una query de SQLAlchemy ahí, todo el servidor web se congela. Ningún otro usuario podrá hacer ninguna petición hasta que termines.

python — Tumbando el Event Loop
import time
from fastapi import FastAPI

app = FastAPI()

@app.get("/error-fatal")
async def error_fatal():
    # ❌ CRÍTICO: time.sleep detiene el EVENT LOOP entero.
    # Si esto tarda 2 segundos, tu servidor atiende 0 peticiones durante esos 2s.
    time.sleep(2) 
    return {"message": "El servidor acaba de revivir"}

❌ Error 2: `await` o librerías Async en funciones `def`

Si usas def (síncrono), FastAPI te pone en un hilo secundario. ¡En ese hilo no hay Event Loop corriendo! Si intentas hacer await, Python te lanzará un SyntaxError. Si llamas a bibliotecas async sin await, recibirás un objeto coroutine indescifrable.

4. Demostración Correcta con TDD

Veamos cómo probar y aislar ambos comportamientos seguros con pytest.

python — test_fastapi_modes.py
from fastapi import FastAPI
from fastapi.testclient import TestClient
import time
import asyncio

app = FastAPI()

# ✅ CORRECTO: Tarea bloqueante en un DEF normal (ThreadPool)
@app.get("/sync-seguro")
def sync_seguro():
    # El Event loop de FastAPI sigue libre atendiendo otras peticiones
    time.sleep(0.5)
    return {"status": "ok"}

# ✅ CORRECTO: Tarea asíncrona en un ASYNC DEF (Event Loop)
@app.get("/async-seguro")
async def async_seguro():
    # Cede el control mientras espera
    await asyncio.sleep(0.5)
    return {"status": "ok"}

client = TestClient(app)

def test_endpoints_respond():
    # Ambos endpoint funcionan y devuelven el resultado esperado.
    # TestClient internamente levanta la app y hace la petición HTTP real.
    assert client.get("/sync-seguro").json() == {"status": "ok"}
    assert client.get("/async-seguro").json() == {"status": "ok"}
