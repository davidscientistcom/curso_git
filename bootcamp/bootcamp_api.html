Evolución y Testing (API Gestión)

Aprende las consecuencias del mal código y cómo refactorizar progresivamente hasta llegar a la Arquitectura Limpia

El Caso Práctico de la Vida Real

En este laboratorio te enfrentamos a un problema común: código heredado de mala calidad. A lo largo de tres fases distintas, verás cómo el código evoluciona y cómo la calidad del testing mejora. Todo esto se encuentra desarrollado en la carpeta bootcamp_api_gestion del repositorio de este curso.

¿Cómo navegar por las Fases?
Cada fase está programada y subida en una rama diferente dentro del repositorio curso_git.
Para acceder al código de cada fase, abre tu terminal y ejecuta: git checkout <nombre_de_la_rama>

Fase 0: El Monolito Inicial

Rama: git checkout 0_fase_inicial

En esta rama te encontrarás un archivo llamado gestion_monolito.py. Contiene la lógica de negocio, validación y almacenamiento de manera completamente "espagueti" (acoplado). Utiliza estado global para las bases de datos y muchísimos if/else que incrementan la complejidad ciclomática.

Problemas y Testing Basura

  1. Dependencias y Herramientas: Para esta fase, necesitas instalar las herramientas de testing en Python. 
    pip install pytest pytest-cov
  2. Fragilidad de los Test: Si haces pytest test_gestion.py -v verás los test. Como el código usa variables globales, ¡el resultado de un test depende de si se ha ejecutado el anterior! Si ejecutas el test de ventas aisladamente, fallará.
  3. Cobertura (Coverage): Ejecuta: 
    pytest --cov=gestion_monolito
    Notarás que la cobertura es bajísima. ¿Por qué? Porque cubrir todas y cada una de las ramas generadas por las validaciones anidadas es un infierno logístico.
  4. El reto: Modifica el código para forzar cambios (como romper un test para ver cómo falla al fallar una validación) e intenta entender por qué el código fuertemente acoplado es doloroso.

Fase 1: Framework y Modularidad

Rama: git checkout 1_fase_fastapi_modular

Hemos desechado el archivo monolítico y lo hemos sustituido por el potente framework web moderno: FastAPI. Ahora el código está dividido en carpetas (Pydantic para Schemas, Rutas, etc.).

Refactorización Intermedia

  1. Instalación: Aquí hay más que pytest. Ejecuta pip install fastapi uvicorn httpx
  2. Cómo solucionamos los problemas de la Fase 0:
    • Hemos delegado TODA la validación masiva de código espagueti a los schemas de Pydantic. Si mandas un email falso, FastAPI devuelve un error de validación 422 Automático.
    • Las rutas están separadas por archivos en la carpeta routers.
    • Testing de APIs Real: Ahora usamos el TestClient. Ya podemos testear la API como un cliente de verdad. Además, hemos configurado Fixtures en Pytest para resetear la base de datos entre tests. ¡El orden ya no importa!

Fase 2: Arquitectura Limpia y SOLID

Rama: git checkout 2_fase_arquitectura_avanzada

La Fase 1 era buena, pero la lógica de la API seguía mezclada con la lógica de base de datos simulada y HTTP. En la versión final, profesionalizamos la estructura organizándolo todo en capas de Clean Architecture (Dominio, Casos de Uso, Infraestructura).

Nivel Senior

  1. Independencia del Framework: Fíjate dentro de src/use_cases. ¡Ahí está toda la lógica contable y de negocio pura, pero no se menciona FastAPI para nada!
  2. Inversión de Dependencias (DIP): FastAPI inyecta las dependencias hacia las interfaces genéricas. Esto permite la genialidad más grande para Testing...
  3. Testing Definitivo (Mocks triviales): Mira el archivo test_api.py. Gracias a app.dependency_overrides, sobreescribir la base de datos para pasarle un "falso repositorio" se hace con 2 líneas de código. 
    # Así sobreescribimos las dependencias para testing
app.dependency_overrides[get_cliente_repository] = lambda: test_cliente_repo
