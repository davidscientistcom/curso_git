Parte 5: Reescribiendo la Historia (Rebase)

Git te permite modificar los commits del historial antes de enviarlos remotamente. Esto no es solo por estética (el OCD del desarrollador); un historial limpio lineal donde cada commit funciona y expone un solo cambio atómico es muy valioso para la empresa al momento de revertir bugs (regression testing o git bisect).

5.1 Un commit que me faltaba (amend)

Hiciste un commit increíble llamado feat: super backend form API endpoint. Segundos después, ves que dejaste un console.log() tirado en el código, o un typo estúpido en el README.

En lugar de hacer otro commit que diga fix: quitar console.log que dejé en el anterior (lo cual ensucia el historial de los demás), puedes "sumar" ese cambio a tu último commit, modificándolo y reescribiéndolo:

# 1. Modifica en el Working directory y borra el console.log
# 2. Haces git add de ese cambio (Staging)
git add server.js

# 3. Y ahora lo engarpas junto con el anterior:
git commit --amend

# Si solo es un cambio pequeño y tampoco querías cambiar el mensaje 
# descriptivo de tu commit anterior, le pones:
git commit --amend --no-edit
ATENCIÓN LA REGLA DE ORO DE GIT: --amend, rebase, y todas estas herramientas cambian el Hash SHA-1 de los commits de forma permanente (los destruyen y crean unos clónicos).

Jamás debes reescribir historia que ya has pusheado y está en manos públicas/colaborativas, destrozarías las ramas de tus compañeros al generar divergencias y conflictos irresolubles. Solo reescribe lo que está local en tu ordenador en tu feature branch.

5.2 Entendiendo Git Rebase

Has sacado tu rama feature-A de main el Lunes. El Martes, Roberto mete unos cambios masivos en main. Tú necesitas esos cambios de Roberto para probar si tu feature-A sigue funcionando o te lo ha roto todo.

En este punto puedes hacer:

A) git merge main (estando en tu rama feature-A). Esto te creará un asqueroso "merge inverso" lleno de cruces en la gráfica.

B) Git Rebase: Arrancar tu rama de donde la creaste e "injertarla" mágicamente por delante de los commits de Roberto.

1. El Escenario Inicial
      A---B---C (main de Roberto)
       \
        X---Y (tu feature-A)
2. Tras ejecutar `git rebase main`
      A---B---C (main)
               \
                X'---Y' (tu nueva feature-A)

(Tus commits antiguos X e Y ahora "rebotan" 
o son "re-baseados" despues de C, con 
nuevos ID Hash)

Conflictos durante un Rebase

El Rebase aplica tus commits X e Y, uno... a... uno... secuencialmente sobre la nueva base. Esto significa que Git puede detenerse y chillar de conflictos múltiples veces (uno por cada commit que está re-baseando).

El flujo cuando te paras por conflicto es ligeramente distinto:

# Estás en feature-A
git rebase main

# - STOP - Conflicto en el commit "X" al reescribirlo.
# Tú vas al código, lo editas, borras los >>><<< y resuelves.
# Y luego le das:
git add fichero-arreglado.txt

# Y en vez de hacer un simple commit como en merge, le dices 'sigue la rueda':
git rebase --continue

Git cogerá eso, cerrará el commit X' resuelto, y avanzará a re-aplicar Y'. Pasa algo? Lo repites. ¿Que te lías y el pánico se apodera? git rebase --abort y tu rama feature-A vieja y sana reaparecerá milagrosamente en el Lunes.

EJERCICIO 7: Rebase Simple (Mudar tu rama)

Objetivo: Rebasear tu features local para traer los cambios de main manteniendo tu código lineal.

Pasos:

  1. Estando en main, asegúrate de estar limpio. Crea un commit: echo "1" > conteo.txt && git add . && git commit -m "main 1"
  2. Saca tu rama paralela: git switch -c feature-math
  3. Haz un par de commits: echo "formula a" > form.txt && git add . && git commit -m "add formula A" echo "formula b" >> form.txt && git add . && git commit -m "add formula B"
  4. Ok, pausa. Vuelve a main y emula a Roberto metiendo cosas críticas. git switch main echo "PARCHE CRITICO" >> conteo.txt && git commit -am "hotfix: parche de seguridad vital"
  5. Tú necesitas ese PARCHE CRÍTICO para que tu rama de matemáticas no pete en preproducción.
  6. Vuelve a tu rama: git switch feature-math
  7. Mágicamente múdate al futuro de main: git rebase main
  8. Visualiza tu historial: git log --oneline --graph. Verás que tu commit de fórmula 2 y 1 están encima del "hotfix parche" de Roberto. Una historia limpia.

5.3 El Superpoder: Rebase Interactivo (-i)

El Rebase interactivo no solo sirve para saltar por encima de otras ramas. Se puede lanzar contra ti mismo (tus últimos 5 commits) para lavar, planchar y comprimir tu historial asqueroso lleno de "WIPs", "Fix", "test3" y convertirlos en una bella joya antes de mergetar a main.

# "Hey Git, quiero que rebases mis últimos 4 commits, y quiero el mando manual"
git rebase -i HEAD~4

Al hacer eso, Git abrirá un editor mostrando tus últimos 4 commits de abajo hacia arriba (más antiguos arriba). Algo así:

pick d458a21 feat: login inicial api
pick 88a31b2 wip: pruebas con el token de jwt
pick 09ab31c fix: ahora si que pilla el token
pick 22bd425 docs: añadir libreria install

# Rebase 22bd425..d458a21 onto d458a21
# Commands:
# p, pick <commit> = use commit
# r, reword <commit> = use commit, but edit the commit message
# e, edit <commit> = use commit, but stop for amending
# s, squash <commit> = use commit, but meld into previous commit
# f, fixup <commit> = like "squash", but discard this commit's log message
# d, drop <commit> = remove commit

El poder está en la palabra "pick". Tú vas a escribir y cambiar el texto (esto es VIM o VS Code) diciéndole a git qué tiene que hacer con la historia si estuviera viajando en el tiempo y reviviéndola:

  • reword (r): Deja el código igual, pero haz que pare la máquina para yo escribir un título de commit de verdad y no "wip xdd".
  • squash (s): Cógelo y en vez de dejarlo como commit puro, amásalo contra el de arriba, fusionándolos. Muy usado para comprimir 4 fix de ortografía o sintaxis a la feature base.
  • drop (d) o simplemente borrar la línea en el editor: Aniquila y extirpa ese commit y sus cambios del multiverso, como si nunca hubieran ocurrido en el sistema (ideal para cuando algo solo rompió y no quieres dejar constancia de la mancha de vergüenza).
En el 99% de los casos usarás la palabra squash contra un pick que esté arriba, fusionando los WIPs en una única "Feat: implementado pasarela stripe super robusta".
