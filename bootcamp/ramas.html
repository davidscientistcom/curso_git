Inicio / 3. Ramas y Merge

Parte 4: Branches (El verdadero poder de Git)

Una rama en Git es simplemente un puntero móvil ligero a un commit específico. Crear una rama es casi instantáneo y solo cuesta unos 41 bytes (40 caracteres del hash y un salto de línea). El concepto de "copiar toda la carpeta" (como hacía SVN) no existe aquí.

4.1 Entendiendo HEAD

HEAD es otro puntero vital. HEAD apunta la rama en la que estás trabajando actualmente. Cuando haces git log ves el historial que va desde lo que está apuntando HEAD hacia atrás.

En Git moderno, cambiar de rama se hace con git switch.
Antiguamente (y todavía se puede) se usaba git checkout para todo (cambiar de rama, descartar archivos, etc.). Switch es más seguro porque hace una sola cosa. 
# Ver todas las ramas locales (* marca la tuya)
git branch

# Crear nueva rama (no te cambia a ella)
git branch nueva-feature

# Cambiar a la rama
git switch nueva-feature

# Crear y cambiar al mismo tiempo (¡SUPER ÚTIL!)
git switch -c fix-bug-login

# Renombrar rama actual
git branch -m feature-login-mejorada

# Eliminar rama (que ya mergeaste)
git branch -d fix-bug-login

# Eliminar a lo bruto (sin preguntar si está mergeada)
git branch -D fix-bug-login

4.2 Feature Branch Workflow

Nunca, y decimos, NUNCA, se desarrolla una nueva funcionalidad (feature) en la rama main (o master). main debería ser la fuente de verdad y una representación del código que está funcionando en Producción en este momento.

El flujo es:

  1. Estás en `main` y está limpio.
  2. Alguien reporta un bug o pide una nueva tabla en la UI.
  3. Haces git switch -c feature/nueva-tabla.
  4. Haces y commiteas todos tus cambios.
  5. Testeas, si se te rompió algo "haces commit --amend". Estás en tu mundo, da igual las roturas.
  6. Cuando funciona: vuelves a main y haces merge de tu feature.

EJERCICIO 5: Crear y Navegar Ramas

Objetivo: Crear diferentes mundos paralelos en tu código sin romper main.

Pasos:

  1. Ve a tu proyecto mi-portafolio.
  2. Verifica tu rama actual (será main).
  3. Crea y cambia a una rama nueva: git switch -c feature-header
  4. Abre index.html, añade una etiqueta <header> y guarda.
  5. Haz commit: git commit -am "feat: agregar banner de header".
  6. La magia de Git: Cambia temporalmente a main con git switch main.
  7. Abre index.html. ¡Sorpresa! El header ha desaparecido. Tu disco duro físico acaba de mutar para reflejar la rama main.
  8. Regresa a tu rama: git switch feature-header. Y tu header volverá a aparecer.

4.3 Merging (Combinando Mundos)

Una vez terminada tu feature o solucionado tu bug en su rama aislada, llega el momento de la verdad.

Para hacer un merge de la "rama B" hacia la "rama A", primero debes estar en la "rama A" (la que recibe) y luego ejecutar el comando merge pidiendo la "rama B".

Tipos de Merge:

1. Fast-Forward (ff)

Ocurre cuando la rama destino (p.ej `main`) no ha avanzado desde que sacaste la feature. Git simplemente "desliza" el puntero de main hacia adelante.

   C1---C2 (main)
         \
          F1---F2 (feat)

git merge feat

C1---C2---F1---F2 (main, feat)
2. Three-Way Merge / Recursive

Ocurre cuando main ha recibido otros commits mientras trabajabas. Git debe crear un nuevo "Merge Commit" (con dos padres) uniendo ambas historias.

   C1---C2---C3 (main)
         \
          F1---F2 (feat)

git merge feat

   C1---C2---C3----M  (main)
         \       /
          F1---F2  (feat)

El merge --no-ff (No Fast Forward)

A veces no queremos un Fast-forward porque queremos dejar constancia de que "esta serie de commits fueron de una feature, y se mergearon juntos". Para forzar la creación del Merge Commit, usamos:

git switch main
git merge --no-ff feature-header

Al hacer tu historial verás el clásico "globo" () que entra y vuelve a salir hacia main, en un `git log --graph`.

4.4 Resolución de Conflictos

Surgen cuando Git va a hacer Merge, pero descubre que la misma línea del mismo archivo ha sido modificada de formas distintas en ambas ramas. Git para el merge y pide a un humano que decida.

git switch main
git merge feature-cambios

# Output de Git:
Auto-merging index.html
CONFLICT (content): Merge conflict in index.html
Automatic merge failed; fix conflicts and then commit the result.

Al abrir tu archivo confictivo (ej. index.html) verás estas infames marcas impresas literalmente en tu código:

<h1>Bienvenido a mi Web</h1>
<<<<<<< HEAD (Tu "main" actual - Donde estás montado)
<p>Este es el texto original rojo.</p>
=======
<p>Este es el nuevo texto verde y azul.</p>
>>>>>>> feature-cambios (La rama que viene y está intentando entrar)

Pasos para resolver un conflicto:

  1. No entres en pánico. git status listará los archivos "Unmerged" (en rojo).
  2. Abre cada archivo en conflicto en tu editor preferido. VSCode las colorea maravillosamente (Choose Current, Choose Incoming, Choose Both).
  3. Borra las de Git (<<<<, ====, >>>>) a mano y deja el código final exactamente como quieres que quede.
  4. Haz git add index.html (esto le dice a Git: «conflicto de index.html resuelto»).
  5. Haz git commit. Como era un merge, Git ya tiene pre-escrito el mensaje "Merge branch feature-cambios", puedes simplemente guardar.

EJERCICIO 6: Provocar y Resolver un Conflicto

Objetivo: Modificar la misma línea en dos ramas y resolverlo a mano sin miedo.

Pasos:

  1. Cambia a main: git switch main
  2. Crea archivo config.txt que diga: COLOR_THEME=blue
  3. Haz add y commit: git add config.txt && git commit -m "feat: tema inicial azul"
  4. Crea rama 'noche': git switch -c tema-noche
  5. Cambia el archivo a: COLOR_THEME=dark y haz commit.
  6. Vuelve a main: git switch main
  7. AQUÍ VIENE LO DIVERTIDO. En main, cambia el mismo archivo a: COLOR_THEME=light y haz commit.
  8. Intenta mergear la rama noche hacia main: git merge tema-noche
  9. ¡Boom! Conflicto. Ejecuta git status para ver el daño.
  10. Abre config.txt, borra las marcas, quédate con una versión lógica o de consenso (ej. COLOR_THEME=dark-light-auto).
  11. Finaliza: git add config.txt && git commit
  12. (Opcional): Revisa el grafo con git log --oneline --graph

4.5 "Abortar, abortar!"

Si estás a mitad de resolver un conflicto enorme que afecta a 50 archivos y te quieres bajar de la vida, no tienes por qué borrar la carpeta. Git tiene un botón del pánico:

# Detiene el merge y devuelve tu Working Directory al 
# último commit seguro de la rama original antes del lío.
git merge --abort
