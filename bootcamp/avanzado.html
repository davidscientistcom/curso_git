Parte 7: Funcionalidades Ninja ("Sácame de este apuro")

A esta altura ya sabes crear y mezclar el 95% de lo que te pedirán en una empresa normal. Pero a veces ocurren accidentes o tienes que encontrar al culpable de un bug fantasma.

7.1 Guardar el trabajo a medias: Git Stash

Estás a la mitad de programar un componente complejo. Te avisa el jefe por un Teams: "Tenemos que subir un parche urgente a la web, apaga lo que tengas y arréglalo en main YA!".

No quieres hacer un commit llamado "wip: asdf" porque rompería e impediría que corrieran los tests del Pipeline CI/CD si se te ocurre pushear algo. Y además está incompleto. Aquí entra git stash (El cajón o bolsillo en español).

# 1. Metes tus ficheros sucios actuales en un bolsillo escondido.
# Esto dejará tu Working Directory totalmente LÍMPIO y vacío como el último commit.
git stash

# Opcionalmente, puedes guardarlos poniéndoles etiqueta para no olvidarte:
git stash push -m "implementando carrito compra, falta CSS"

# --- AHORA SALES, ARREGLAS EL MAIN, HACES PUSH COMO TE PIDIERON ---

# 2. Vuelves a tu rama de la feature vieja e invocas el bolsillo de vuelta.
# pop saca los datos en tus archivos MIENTRAS borra la entrada del bolsillo.
git stash pop

# Ver todo lo que tienes guardado en múltiples bolsillos
git stash list
Si haces git stash apply los cambios volverán a tu fichero, pero el estado guardado no se borrará del listado del stash, útil por si crees que algo se puede romper al re-aplicar y quieres mantener el backup.

7.2 Robar una característica: Cherry Pick

A veces no quieres fusionar toda la rama entera de un programador (porque tiene 20 features y bugs). Solo quieres ese commit exacto de esa línea que parseaba el JSON porque la necesitas para tu desarrollo paralelo.

# Encuentras el hash del commit al ver en GitHub o en el log ajeno (ej. e2f9a1)
# Y te lo llevas mágicamente como un "copy-paste de cambios" por encima a tu rama:

git cherry-pick e2f9a1c

7.3 Encontrar la Aguja en el Pajar: Git Bisect

El botón de login se ha roto. Estás seguro que en Producción (versión v1.0.0 de hace 3 semanas) funcionaba perfectamente. Hoy (HEAD) estalla de error 500. Ha habido 140 commits desde esa versión. ¿Quién coj**** lo rompió y qué línea puso?

En vez de revisar uno a uno como un loco, usarás la Búsqueda Binaria automatizada:

# Iniciamos el motor bisect (El buscador)
git bisect start

# A) Le marcas el límite "MALO" (El de hoy que sabes que está roto)
git bisect bad

# B) Le marcas el límite "BUENO" (Un hash o Tag de hace un mes que sabes iba joya)
git bisect good v1.0.0

# OUTPUT MAGICO DE GIT:
# "Bisecting: 70 revisions left to test after this (roughly 6 steps)"

# Git te ha transportado EN MEDIO DEL HISTORIAL y ha compilado el código.
# Tú pulsas en el botón de login. "Oh funciona!".
git bisect good

# Git salta de nuevo, partiendo la historia. Tú pulsas de nuevo. "Ouch, Fallo"
git bisect bad

# Tras unos pocos clics, el Asesino revelado:
# <e13a9bf0> is the first bad commit
# Author: Roberto D.<roberto@mail.com>
# feat: optimizar authService inyectando caché roto

# Acabas y devuelves a git a como estaba hoy pacíficamente
git bisect reset

EJERCICIO 9: El Salvavidas Definitivo (Git Reflog)

Objetivo: Rescatar un commit borrado o una rama que has ejecutado con -D sin querer.

En Git NADA SE BORRA, CASI NADA (salvo que corra el garbage collection pasados más de 30 días, o archivos no trackeados nunca que borras a tu nivel de disco). Git guarda un registro diario, como una "Caja Negra de Avión", de todo movimiento que haces por minúsculo que sea.

Pasos:

  1. En un repositorio cualquiera, haz dos commits limpios. Reemplaza palabras simples y guarda (ej: "A", luego "B").
  2. Hagamos una catástrofe. Resetea duramente volviendo al penúltimo commit como si B no existiera.
    git reset --hard HEAD~1
  3. Haz un git log. Te sudarán las manos. B ha desaparecido para siempre. La rama no existe, el cambio se fue no está ni trackeado ni subido al remoto.
  4. Usa la máquina del tiempo:
    git reflog
  5. Deberías ver una especie de bitácora extraña de acciones como:
    9a3b2f HEAD@{0}: reset: moving to HEAD~1
    ec4b7a HEAD@{1}: commit: texto B <-- ¡AQUÍ ESTÁ! Este es el hash criptográfico huérfano y perdido.
  6. Si quieres viajar atrás y crear urgentemente una rama cogiendo ese hash zombi del éter, solo escribe:
    git branch salvavidas ec4b7a
  7. El código de B ha resucitado como si la catástrofe nunca hubiera ocurrido en la nueva rama salvavidas.
Ojo: Al invocar un --hard, si de veras perdiste horas de trabajo en el Working Directory que JAMÁS COMPRIME EN UN COMMIT NI EN STAGE, de ninguna manera; eso de verdad no lo rescata ni la CSI Forense. Nunca borres cosas a lo bestia en tu disco. De lo contrario, con commit, nada se pierde en Git.
