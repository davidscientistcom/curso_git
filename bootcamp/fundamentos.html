Inicio / 1. Fundamentos

Parte 1: Instalación y Configuración Base

Antes de empezar a usar Git, es esencial configurarlo correctamente. Git necesita saber quién eres para poder asociar tus commits a tu identidad.

1.1 Instalación Rápida

Dependiendo de tu sistema operativo, la instalación varía:

Linux (Debian/Ubuntu)
sudo apt update
sudo apt install git
macOS
brew install git

O instalando Xcode Command Line Tools: xcode-select --install

Windows

Descarga el instalador oficial desde git-scm.com. Recomendamos usar Git Bash (incluido en la instalación) en lugar de CMD o PowerShell para tener una experiencia UNIX-like uniforme con este curso.

1.2 Configuración Inicial (Esencial)

Existen tres niveles de configuración en Git:

  • --system: Para todos los usuarios del sistema (raro).
  • --global: Para tu usuario en todos los repositorios (lo más común). ~/ .gitconfig
  • --local: Solo para el repositorio actual (sobrescribe global). .git/config
bash 
# Configurar nombre y email (Obligatorio)
git config --global user.name "Tu Nombre"
git config --global user.email "tu@email.com"

# Configurar el editor por defecto (Recomendado)
# Para VS Code:
git config --global core.editor "code --wait"
# Para Vim:
git config --global core.editor "vim"
# Para Nano:
git config --global core.editor "nano"

# Configurar el nombre de la rama principal a 'main'
git config --global init.defaultBranch main

# Ver tu configuración actual
git config --list

EJERCICIO 1: Configurar el Entorno

Objetivo: Dejar Git listo con tu identidad y editor preferido.

Pasos:

  1. Abre tu terminal.
  2. Verifica que Git está instalado: git --version
  3. Configura tu nombre real: git config --global user.name "Tu Nombre Completo"
  4. Configura tu email (usa el mismo de tu cuenta de GitHub): git config --global user.email "tu@email.com"
  5. Configura tu editor favorito (ejemplo VS Code): git config --global core.editor "code --wait"

Resultado esperado:

Al ejecutar git config --global -e debería abrirse tu editor con un archivo que contiene la configuración que acabas de introducir. Ciérralo sin modificar.

Parte 2: Cómo Funciona Git por Debajo

A diferencia de SVN o CVS (que guardan deltas/diferencias de archivos), Git piensa en sus datos más como un sistema de archivos snapshot stream.

Cada vez que haces un commit, Git toma una foto (snapshot) de cómo se ven TODOS tus archivos en ese momento y guarda una referencia a esa foto. Para ser eficiente, si los archivos no han cambiado, Git no almacena el archivo de nuevo, solo un enlace al archivo idéntico que ya tiene almacenado.

2.1 El Modelo de Tres Áreas

Este es el concepto más importante en Git. Un archivo en Git puede residir en tres áreas principales:

Working Directory Tus archivos actuales

Donde modificas el código

Staging Area (Index) Archivos marcados

Preparados para el próximo commit

Local Repository Directorio .git

Base de datos de Git permanente

Por lo tanto, un archivo puede estar en uno de cuatro estados:
  • Untracked: Archivos nuevos que Git no está vigilando.
  • Unmodified: Archivos vigilados que no han cambiado desde el último commit.
  • Modified: Archivos vigilados que has modificado en tu Working Directory.
  • Staged: Archivos modificados/nuevos que has marcado para que vayan en tu próximo commit.

2.2 Los Objetos de Git

Bajo la carpeta .git/objects, Git almacena todo como una base de datos clave-valor. Todo tiene un hash SHA-1 de 40 caracteres con el que se identifica. Los 4 tipos de objetos son:

Blob

Almacena el contenido de un archivo. No guarda el nombre del archivo ni sus permisos, solo la data cruda.

Tree

Representa un directorio. Contiene listas de punteros a Blobs (archivos) u otros Trees (subdirectorios), junto con sus nombres y permisos.

Commit

Es la "fotografía" temporal. Apunta a un Root Tree (estado del proyecto). Contiene autor, fecha, el mensaje de commit y un puntero a su commit Padre.

Tag

Un puntero inamovible (usualmente a un Commit) que incluye un nombre humanamente legible (ej. "v1.0.0"), un mensake y quién lo creó.

EJERCICIO 2: Inspeccionar los Internals

Objetivo: Ver con tus propios ojos cómo Git crea y almacena objetos internamente. No harás esto en el día a día, pero te dará "superpoderes" para entender Git.

Pasos:

  1. Crea una carpeta de prueba vacía y entra en ella: mkdir test-git && cd test-git
  2. Inicia un repo vacío: git init
  3. Observa el contenido de la carpeta .git/objects (debería estar vacía excepto por info y pack): ls -la .git/objects
  4. Escribe un string directamente a la base de datos de Git creando un Blob: echo "Hola Mundo" | git hash-object -w --stdin (Verás que devuelve un hash como: 557db03de997c86a4a028e1ebd3a1ceb225be238)
  5. Vuelve a listar la carpeta objects. Verás una nueva carpeta con los dos primeros caracteres del hash: ls -la .git/objects
  6. Lee el contenido del objeto que acabas de crear usando su hash: git cat-file -p {reemplaza-con-el-hash-devuelto}
  7. Verifica el tipo de objeto (debería decir "blob"): git cat-file -t {reemplaza-con-el-hash-devuelto}

Resultado esperado:

Entiendes que Git funciona como una simple base de datos "llave-valor" donde la llave es el hash criptográfico SHA-1 y el valor es el contenido comprimido.
