Parte 3: El Día a Día (Operaciones Básicas)

Ahora que entiendes el Staging Area, vamos a ver el flujo real diario de un desarrollador.

3.1 Crear un Repositorio (git init)

Para empezar a usar Git en tu proyecto, solo tienes que situarte en la carpeta y ejecutar:

cd mi-proyecto
git init

Esto crea una subcarpeta oculta .git/ donde se almacenará todo el historial. Tu carpeta ahora es un "Working Directory" válido.

3.2 Comprobar el Estado (git status)

Este es el comando que más usarás. Te dice en qué estado se encuentran tus archivos (Untracked, Modified, Staged).

git status

# Forma corta y resumida (muy recomendada):
git status -s  # Muestra ?? (untracked), M (modified), A (added), etc.

3.3 El Staging Area (git add)

git add escoge archivos de tu Working Directory y los copia al Index (Staging Area). Piensa en esto como "preparar la caja" antes de cerrarla y enviarla.

# Añadir un archivo específico
git add index.html

# Añadir varios archivos o una carpeta entera
git add css/ src/ index.html

# Añadir TODO el directorio actual (lo más usado)
git add .

# Añadir de forma interactiva (chunk por chunk del archivo)
git add -p
Pro Tip: git add -p (patch module) es increíblemente útil cuando has hecho demasiados cambios en un archivo pero quieres dividir esos cambios en diferentes commits lógicos.

3.4 Empaquetar el Historial (git commit)

Una vez la caja (Staging Area) tiene lo que quieres, la "cierras" y le pones una etiqueta descriptiva: eso es el commit.

# Abrirá tu editor configurado para escribir el mensaje
git commit

# Mensaje corto directo en consola
git commit -m "add login page validation"

# Atajo: hace 'git add' de todos los archivos YA traqueados y commitea
# Nota: No añadirá archivos nuevos (untracked)
git commit -am "fix header layout shift on mobile"

Buenas Prácticas en los Mensajes de Commit

Un historial lleno de "updates", "wip", o "fix cosas" es inútil cuando buscas un bug 6 meses después. Usa la convención de Conventional Commits.

Tipo Cuándo usarlo Ejemplo
feat Una nueva característica funcional feat: añadir validación de contraseña en registro
fix Solución de un bug o error fix: botón de submit deshabilitado tras error 500
refactor Cambio de código que no añade feat ni fix refactor: extraer lógica de auth a middleware
docs Cambios en documentación o comentarios docs: actualizar README con pasos de instalación
style Formateo, punto y coma, tabs (no cambia lógica) style: aplicar prettier formatting en componentes
test Añadir o corregir pruebas unitarias test: cubrir casos límite del carrito de compras

EJERCICIO 3: El Flujo Diario

Objetivo: Crear un proyecto desde cero, añadir archivos, verificar el estado y hacer commits limpios.

Pasos:

  1. Crea un proyecto: mkdir mi-portafolio && cd mi-portafolio
  2. Inicializa Git: git init
  3. Crea un archivo de prueba (Linux/Mac): echo "<h1>Hola Mundo</h1>" > index.html
  4. Verifica estado: git status (Verás que es Untracked).
  5. Añade al Stage: git add index.html
  6. Verifica de nuevo: git status (Verás que está listo para commit).
  7. Haz el commit inicial: git commit -m "feat: inicializar portafolio con index vacio"
  8. Modifica el archivo: Abre index.html y añade contenido.
  9. Crea un segundo archivo: style.css.
  10. Verifica estado: git status (Verás uno Modified y otro Untracked).
  11. Añade todo y haz commit: git add . && git commit -m "feat: agregar estilos basicos y contenido a index"

3.5 Historial y Cambios

Ver el historial (git log)

# Historial estándar (largo y detallado)
git log

# ¡SUPER ÚTIL! Historial compacto en una línea
git log --oneline

# El "log ninja": Visualización gráfica del historial de ramas (graph)
git log --oneline --graph --all

Ver modificaciones (git diff)

# Diferencias entre archivos locales vs Staging Area
git diff

# Diferencias entre Staging Area vs último commit (¿Qué vas a commitear?)
git diff --staged

# Qué cambios introdujo exactamente el último commit u otro específico
git show HEAD
git show 5f8a9e2

EJERCICIO 4: Explorar el Historial

Objetivo: Aprender a navegar por los cambios que has hecho.

Pasos:

  1. En la carpeta mi-portafolio, ejecuta: git log --oneline
  2. Añade un párrafo a index.html sin hacer git add todavía.
  3. Ejecuta git diff para ver exactamente qué líneas cambiaste.
  4. Haz git add index.html.
  5. Vuelve a ejecutar git diff. ¡El resultado estará vacío! (porque el archivo ya pasó al stage).
  6. Ejecuta git diff --staged para ver esos cambios listos para el commit.
  7. Haz el commit: git commit -m "docs: extender parrafo introductorio".

3.6 Archivos Ignorados (.gitignore)

Hay archivos que NUNCA deberían subirse a Git (variables de entorno, claves secretas, dependencias pesadas como node_modules, binarios compilados, archivos de tu IDE como .vscode).

Para eso se crea un archivo llamado .gitignore en la raíz.

.gitignore 
# Ignorar carpetas Node
node_modules/

# Ignorar archivos ocultos del sistema operativo
.DS_Store

# Ignorar logs
*.log

# Ignorar variables de entorno
.env

# Excepción: ignorar todos los .txt, EXCEPTO roadmap.txt
*.txt
!roadmap.txt
Si alguna vez haces git add sobre un archivo y le haces commit, añadirlo al .gitignore después no lo borrará de Git. Seguirá siendo vigilado. Veremos cómo solucionar esto en la sección de Deshacer Cambios.
