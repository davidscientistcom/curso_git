Paso 1 de 6 Instalar la extensión de Python en VS Code

Antes de cualquier cosa, necesitas la extensión oficial de Microsoft. Esta extensión es la que conecta VS Code con el intérprete de Python y habilita el depurador.

Abre VS Code y ve al panel de extensiones con Ctrl+Shift+X. Busca Python y asegúrate de instalar la de Microsoft (el autor debe ser "Microsoft").

Nota: Con la extensión Python también se instala automáticamente la extensión Pylance para autocompletado y análisis de tipos. Ambas trabajan juntas.
  • Extensión "Python" de Microsoft instalada
  • VS Code reconoce el intérprete Python del proyecto
Paso 2 de 6 Estructura del proyecto y entorno virtual

VS Code necesita saber qué intérprete usar. Lo más limpio es tener un entorno virtual dentro del proyecto. Si ya tienes uno, selecciónalo con Ctrl+Shift+P → Python: Select Interpreter.

Una estructura típica que funciona bien con pytest sería:

mi_proyecto/ ├── src/ │ └── calculadora.py # tu código fuente ├── tests/ │ └── test_calculadora.py # tus tests pytest ├── .venv/ # entorno virtual └── pyproject.toml # o requirements.txt
Importante: pytest busca tests en ficheros que empiecen por test_ o terminen en _test.py. Mantén esta convención o el depurador no los detectará.

Si no tienes entorno virtual aún, créalo desde el terminal integrado:

python -m venv .venv source .venv/bin/activate pip install pytest
Paso 3 de 6 Configurar pytest como framework de tests

VS Code necesita saber que usas pytest (no unittest, que es el predeterminado). Ve a Ctrl+Shift+P → Python: Configure Tests y selecciona pytest. Elige la carpeta raíz del proyecto.

Esto crea automáticamente (o modifica) el fichero .vscode/settings.json:

.vscode/settings.json
{ "python.testing.pytestEnabled": true, "python.testing.unittestEnabled": false, "python.testing.pytestArgs": [ "tests" // carpeta donde están tus tests ] }
Tip: Si tus tests están en la raíz o en una subcarpeta diferente, cambia el valor de pytestArgs para que apunte a ella. Puedes añadir opciones de pytest aquí, como "-v" para verbose.
Paso 4 de 6 Crear el fichero launch.json para el depurador

Este es el paso clave. El fichero launch.json le dice a VS Code cómo lanzar el depurador. Sin él, solo puedes ejecutar tests pero no pararlos en un breakpoint.

Ve a Ctrl+Shift+D (panel Run and Debug) → create a launch.json file → selecciona Python Debugger → Python: Remote Attach o simplemente edita el fichero a mano:

.vscode/launch.json
{ "version": "0.2.0", "configurations": [ { "name": "Debug Tests (pytest)", "type": "debugpy", "request": "launch", "module": "pytest", // lanzamos pytest como módulo "args": [ "tests/", // ruta a tus tests "-v", // verbose: ves qué test se ejecuta "--no-header", "-rN" ], "console": "integratedTerminal", "justMyCode": true // false si quieres entrar en librerías } ] }
¿Por qué "module": "pytest" y no "program"? Con module usamos el pytest del entorno virtual activo, evitando problemas de path. Es la forma recomendada en Linux.
Paso 5 de 6 Poner breakpoints y lanzar la depuración

Ahora lo interesante. Abre cualquier fichero de test o de código fuente y haz clic en el margen izquierdo junto al número de línea para poner un breakpoint (aparece un punto rojo).

Para lanzar el depurador tienes dos caminos:

  • Desde el panel Test Explorer (icono de matraz en la barra lateral): busca el test que quieres depurar, haz clic derecho → Debug Test. Solo ejecuta ese test.
  • Desde Run and Debug F5: usa la configuración launch.json y ejecuta todos los tests con el depurador activo.
Atención: Si usas Ctrl+F5 (Run Without Debugging) o el botón de play del Test Explorer normal, el depurador no se activa y los breakpoints se ignoran.

Una vez detenido en un breakpoint, puedes inspeccionar variables en el panel Variables, evaluar expresiones en Watch, y navegar con:

F10 → Step Over (ejecuta la línea sin entrar en funciones) F11 → Step Into (entra dentro de la función llamada) F5 → Continue (continúa hasta el siguiente breakpoint)
Paso 6 de 6 Depurar un test concreto o con parámetros

Cuando tienes muchos tests, relanzar todos es lento. Puedes configurar el launch.json para depurar solo un test específico, o usar el Test Explorer para ir directamente al que falla.

Depurar solo un test por nombre
"args": [ "tests/test_calculadora.py::test_suma", // solo este test "-v" ]
Depurar tests que coincidan con un patrón
"args": [ "-k", "suma or resta", // tests cuyo nombre contenga "suma" o "resta" "-v" ]

Para tests parametrizados con @pytest.mark.parametrize, el depurador se detendrá en el breakpoint en cada iteración del parámetro, lo que te permite ver exactamente qué valor está fallando.

Tip final: Si justMyCode: true en el launch.json y quieres entrar dentro de una fixture o librería de terceros, cámbialo a false. Recuerda que con Hypothesis los tests se ejecutan muchas veces — pon el breakpoint dentro del cuerpo del test, no en la firma.
