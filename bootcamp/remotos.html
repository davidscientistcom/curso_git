Parte 6: Repositorios Remotos (GitHub, GitLab)

Hasta ahora, todo nuestro trabajo ha sido local. Pero el poder real de Git reside en su naturaleza distribuida. Puedes tener múltiples copias (remotes) de tu proyecto sincronizadas en la nube o en servidores locales.

6.1 ¿Qué es un Remote y qué es Origin?

Un "remote" (remoto) en Git es simplemente una URL con un alias (un nombre corto). Te evita tener que escribir la URL completa de GitHub cada vez que quieres enviar código.

Por convención universal, cuando clonas un repositorio por primera vez, Git le asigna automáticamente el nombre origin a la URL de donde lo descargaste.

# Clonar un repositorio existente (crea origin automático)
git clone https://github.com/usuario/repo.git

# Listar los remotos configurados
git remote -v
# Output esperado:
# origin  https://github.com/usuario/repo.git (fetch)
# origin  https://github.com/usuario/repo.git (push)

# Si empezaste en local (git init), añadir un nuevo remoto:
git remote add origin https://github.com/usuario/repo.git
Ojo: GitHub, GitLab o Bitbucket NO son Git. Son plataformas de alojamiento que tienen instalado Git en sus servidores y te proveen una interfaz gráfica web para visualizarlo y gestionar accesos (Pull Requests y Issues no existen en Git clásico, son inventos de GitHub).

6.2 Push, Fetch y Pull

git push

Envía tus commits locales al servidor remoto.

# Subir tu rama main al remote origin
git push origin main

# Si es la PRIMERA VEZ que subes una rama nueva recién creada,
# necesitas el flag -u (--set-upstream) para enlazar ambos mundos
git push -u origin feature-perfiles
# ¡A partir de ahora, bastará solo con teclear 'git push'!
git fetch (Descarga Segura)

Descarga la información nueva del remoto a tu PC (ramas de otras personas, commits en main), pero NO toca tu Working Directory. Se queda guardado en la recámara.

git fetch origin
git pull (Descarga Destructiva)

Es un atajo de dos comandos por debajo: git fetch + git merge. Se descarga lo nuevo e intenta fusionarlo activamente sobre los archivos que tienes abiertos.

git pull origin main
Error al descargar si tu historial local está sucio: Si tienes archivos modificados y sin commitear en tu editor y lanzas un comando de git pull (especialmente si tu Git está configurado para hacerlo usando rebase por defecto), te bloqueará con este mensaje:

error: no se puede pull con rebase: Tienes cambios sin marcar.
error: Por favor, confírmalos o guárdalos.

Se trata un bloqueo intencionado. La descarga con fusión sobreescribiría los cambios en los que estás trabajando actualmente. La solución es limpiar tu árbol de trabajo antes de traerte nada: Puedes hacer un git commit normal para sellar tus archivos, o si estaban a medias, puedes "guardarlos temporalmente en un cajón" haciendo git stash, luego lanzas tu git pull y tras descargar haces git stash pop para recuperar todo como lo tenías.

6.3 Resolviendo Conflictos en el Remoto (Pull Requests)

Imagina que intentas hacer git push y Git te escupe un error gigante y rojo diciendo "REJECTED (non-fast-forward)". ¿Qué ha pasado?

Ha pasado que un compañero de equipo hizo un push a main media hora antes que tú. El remeto "sabe cosas" que tu PC no sabe. Git no te dejará sobreescribir la historia.

La Solución (El Pull Resolutivo):

  1. Git te rechazó el push.
  2. Haces git pull origin main.
  3. Si tocasteis las mismas líneas, saltará un CONFLITO DE MERGE local en tu pantalla.
  4. Abres el archivo, eliges Accept Current o Accept Incoming y guardas.
  5. Haces git commit y cierras el conflicto.
  6. Vuelves a hacer git push, con la historia fusionada, y GitHub lo aceptará.

EJERCICIO 8: Simulacro de Colaboración

Objetivo: Comprender el proceso de clonar, crear ramas asociadas (tracking) y empujar.

Pasos:

  1. Crea un repositorio vacío en GitHub (sin README).
  2. Copia su URL HTTPS o SSH (ej. https://github.com/usuario/test-repo.git)
  3. Vuelve a un proyecto tuyo local (mi-portafolio) que hicimos antes.
  4. Vincula el remoto: git remote add origin la_url_que_copiaste
  5. Empuja tu historia base a la nube: git push -u origin main
  6. Crea un problema adrede: Entra a GitHub desde la web, edita un archivo directamente allí y dale a commit. Esto imita a "un colega ha cambiado cosas y tú no te has enterado".
  7. Vuelve a tu PC local, intenta editar el mismo archivo en el mismo sitio.
  8. Haz git commit -am "Cambio en local rompiendo cosas".
  9. Intenta git push. ¡Te dará un ERROR (Rejected)!
  10. Haz lo que dijimos: git pull origin main, resuelve el conflicto pacíficamente en tu editor y luego vuelve a intentar el git push. ¡Victoria!

6.4 Limpieza: Las Ramas Antiguas

Un error muy común de junior es no limpiar después de que te aprueban la Pull Request (PR). La rama sigue existiendo localmente y remotamente.

# 1. Cuando te aprueben en GitHub, borra su rama remota 
# O le das al botón morado "Delete branch" en la web.

# 2. Te sincronizas y limpias la basura de ramas remotas muertas de tu PC:
git fetch --prune

# 3. Y por fin borras la local que ya no vale para nada:
git switch main
git branch -d feature-login-terminado
