Prueba Final: De Cero a Git Master

Este laboratorio está diseñado para poner a prueba absolutamente todo lo que has aprendido. No te daremos los comandos exactos masticados paso a paso. Te daremos los requisitos y escenarios, y tú deberás usar tu terminal para resolverlos. Te recomendamos dedicar varias horas a este ejercicio, rompiendo cosas sin miedo y solucionándolas.

Aviso: Si te atascas en algún punto, vuelve a leer las lecciones correspondientes. No busques ciegamente en Stack Overflow copiando comandos que no entiendes. Usa `git status` y `git log --oneline --graph --all` constantemente para guiarte.

Fase 1: El Despegue (Fundamentos y Operaciones)

Imagina que te acaban de contratar en una startup para rehacer su plataforma principal. Te han dado acceso a tu nueva máquina y te piden que empieces a estructurar el proyecto.

Misión 1

  1. Crea un directorio llamado startup-platform e inicializa un repositorio de Git completamente vacío.
  2. El sistema va a generar archivos basura de logs (todo lo que termine en .log) y secretos (un archivo llamado secrets.env). Crea la estructura necesaria usando .gitignore para que Git jamás mire esos archivos.
  3. Crea un archivo index.js que contenga un simple console.log("Iniciando sistema");.
  4. Haz que tu primer commit incluya tanto el archivo ignorador como el script inicial, usando la convención correcta de Conventional Commits (ej. algo tipo `feat` o `init`).
Ver Validación Esperada
Al ejecutar git status tu árbol debe estar limpio. Si creas manualmente por terminal un archivo error.log y otro secrets.env e intentas hacer git add ., Git debería ignorarlos por completo sin incluirlos en el Stage.

Fase 2: El Desarrollo en Paralelo (Ramas y Merge)

El proyecto arranca y hay dos tareas que hacer a la vez. Como eres "Full Stack", vas a hacer ambas, pero en mundos aislados para no cruzar cables.

Misión 2

  1. Crea y muévete a una rama llamada feature/login. En esa rama edita tu index.js añadiendo la lógica para un login ficticio y haz un commit limpio.
  2. Vuelve a la rama main (asegúrate de que el login desaparece de tu editor).
  3. Crea y muévete a otra rama llamada feature/database (¡Asegúrate de haberla sacado desde main y no desde login!). Añade un archivo db.json, ponle unos datos falsos y haz un commit en esa rama.
  4. Integra todo: vuelve a main. Haz merge de la rama de base de datos. Como no hay divergencias, fíjate que hará un Fast-Forward.
  5. Ahora haz merge de la rama de login. Como tu main ya avanzó gracias a la base de datos, este merge forzará un Three-Way Merge. Escribe un mensaje de commit bonito cuando Git te abra el editor.
  6. Elimina ambas ramas de feature, ya que no sirven de nada.
Ver Validación Esperada
Al ejecutar git log --oneline --graph, deberías ver una línea recta al principio, una bifurcación que sale y luego un globo que vuelve a unirse a la rama principal (un "merge commit").

Fase 3: El Desastre Inminente (Time Travel y Reset)

Eran las 3 de la mañana y cometiste un error gravísimo. El CTO está a punto de revisar tu código.

Misión 3

  1. Estando en main, edita el index.js y pon tu contraseña personal del banco en texto plano (ej. const db_pass = "123456";). Haz un commit llamado fix: añadidas contraseñas seguras.
  2. Te das cuenta del error. Quieres matar ese commit como si nunca hubiera existido en la historia (como no lo has subido a GitHub aún, es totalmente lícito asesinar el historial). Usando la máquina del tiempo de Git (git reset --hard), vuelve exactamente al commit anterior, eliminando tu commit de las contraseñas para siempre.
  3. Revisa tu código, la contraseña ha desaparecido mágicamente de tu editor.
  4. ¡Espera! Resulta que en ese commit también habías añadido 300 líneas de código vital que ahora se han esfumado porque borraste a lo bruto. Misión crítica: ¡Resucita el código muerto!
  5. Ejecuta git reflog, encuentra el hash criptográfico del commit que acabas de destruir sin piedad.
  6. Crea una rama llamada rescate a partir de ese hash perdido y muévete a ella para recuperar tus archivos.

Fase 4: El Caos Colaborativo (Conflictos en Remoto)

Para esto, vamos a simular a dos programadores compartiendo repositorio, clonando la carpeta en dos sitios distintos en tu PC local.

Misión 4

  1. Vuelve a main en tu primera carpeta. Inicializa un repositorio "Bare" (sin working directory) falso como si fuera el servidor de GitHub en una carpeta separada por completo: cd .. && git init --bare servidor.git
  2. En tu primera carpeta, engancha ese servidor ficticio: git remote add origin ../servidor.git y empuja main: git push -u origin main
  3. ¡Simula a tu compañero Roberto! Vete a otro sitio de tu PC, ponte en otra carpeta y haz git clone la_ruta_a_servidor.git roberto-folder.
  4. Como "Roberto", edita la primera línea del archivo index.js. Haz un commit y haz git push al servidor.
  5. Vuelve a tu carpeta original (Tú). SIN HABER HECHO PULL TODAVÍA, edita exactamente la misma primera línea del archivo index.js con otra cosa.
  6. En lugar de hacer commit, haz algo a medias. Usa git stash para guardártelo en el cajón temporal.
  7. Haz git pull de lo que subió Roberto. Verás que entra suavemente.
  8. Saca tu trabajo del cajón: git stash pop. ¡BAM! Conflicto inminente contra el código de Roberto.
  9. Abre tu editor, localiza los <<<<, ==== y >>>>, borra las marcas y quédate con una versión que junte ambas cosas lógicamente. Haz add, commit final y push. Has ganado.

Fase 5: Cirugía Mayor y Estética (Rebase Interactivo)

A tu Jefe Técnico (Tech Lead) no le gustan los historiales sucios. Y tú, por desgracia, has estado trabajando muy desordenadamente.

Misión 5

  1. Saca una nueva rama: git switch -c feature/dashboard
  2. Comienza a cambiar varias cosas del proyecto creando commits MUY MALOS:
    • Borra una línea, haz commit: "wip 1"
    • Añade un botón, haz commit: "test"
    • Arregla un typo del botón, haz commit: "fix botón tonto"
    • Termina el diseño, haz commit: "ya funciona casi"
  3. Ahora tienes 4 commits espantosos. Antes de hacer merge a main, ejecuta Git Rebase Interactivo diciéndole a Git que afecte a esos últimos 4 commits (HEAD~4).
  4. Se abrirá tu terminal. Cambia la palabra pick de los 3 últimos commits por la palabra squash (o la letra s), para "aplastarlos" hacia arriba contra el primer commit.
  5. Deja el primer commit que dice "wip 1" con la palabra reword (o r) para poder reescribir su título.
  6. Guarda y cierra el editor. Git te pedirá inmediatamente un nuevo título consolidado. Escribe: feat: añadir dashboard completo de estadísticas.
  7. Revisa tu git log --oneline. Magia. Tienes un historial glorioso, inmaculado, de un solo commit profesional. Ya puedes volver a main y mergearlo orgullosamente.
Enhorabuena
Si has sido capaz de resolver todos estos problemas tú mismo sin destruir irremediablemente ningún repositorio, oficialmente eres un experto en Git con más capacidad real de resolución de conflictos que el 70% de la industria. ¡Estás listo para usar GitHub a nivel profesional!
